## H100x8 에서 로그로 확인되는 CPU pinning 범위

boot log 기준으로 `054010` 과 `044922` 모두 CPU engine 이 다음 `local_omp_cpuid` 범위를 사용한다.

- NUMA 0 engine: `112..167`
- NUMA 1 engine: `168..223`

여기서 로그만으로 확정 가능한 것은 다음 두 가지다.

- CPU worker 가 NUMA-aware 2-engine 구조로 올라왔다는 점
- 각 engine 이 위 범위의 CPU id 집합을 pinning 대상으로 사용했다는 점

반면 로그만으로는 다음을 단정할 수 없다.

- 위 CPU 집합이 topology 상 `physical` 인지 `HT sibling` 인지
- 실제 utilization 분포가 어느 절반에서 더 높았는지

즉 이 문서에서는 CPU pinning 대상을 `local_omp_cpuid` 로그 범위까지만 기록하고, 그 하드웨어 해석은 별도 근거 없이는 확장하지 않는다.

## 왜 CPU tail 이 생기는가

현재 구조는 request-level partition 이다.

- GPU request 는 GPU 가 끝까지 책임진다.
- CPU request 는 CPU 가 끝까지 책임진다.
- 중간 handoff 없음

코드 경로:

- add_request: `/vllm_hybrid/vllm/v1/engine/core_client.py`
- route: `/vllm_hybrid/vllm/v1/engine/hybrid_core.py`
- finish callback: `process_engine_outputs()` -> `on_request_finished()`

이 구조에서 wall time 은 사실상:

- `max(T_gpu_bulk_done, T_cpu_tail_done)`

가 된다.

H100x8 7B 128/128 burst 에서는 GPU가 매우 빨라서 `T_gpu_bulk_done` 는 수 초 수준이고, CPU tail 이 항상 지배한다.

그래서:

- `max_seqs=1` -> CPU tail 2개
- `max_seqs=16` -> CPU tail 32개

가 그대로 wall time 에 반영된다.

## wave-batch engine 선택 세부

`wave-batch` 에서 CPU engine 선택은 단순 round-robin 이 아니라 현재 wave 의 `accepted` 수가 더 적은 engine 을 우선 선택한다.

코드:

- `/vllm_hybrid/vllm/v1/engine/hybrid_core.py`
- `_find_wave_open_cpu()`

핵심 비교:

```python
if self._cpu_wave_accepted[i] < best_accepted:
    best_accepted = self._cpu_wave_accepted[i]
    best_idx = i
```

의미:

- 두 engine 의 accepted 가 다르면 더 적게 받은 쪽으로 보낸다.
- accepted 수가 같으면 먼저 본 engine index 가 유지된다.

실제 동작 결과:

- `max_seqs=1` 에서는
  - 첫 main req -> engine 0
  - 둘째 main req -> engine 1
  - 그 뒤 둘 다 closed
- `max_seqs=16` 에서는
  - engine 0, engine 1 이 accepted 수를 맞춰가며 사실상 alternating 하다가
  - 각 engine 이 16개씩 받아 총 32개에서 closed 된다.

따라서 `045947` 의 CPU 32 req 는 “한 engine 이 32개를 독식”한 게 아니라 “두 engine 이 각 16개씩 wave 를 채운 결과”로 해석하는 것이 맞다.

## `044922` vs `054010` 비교에서 주의할 점

겉으로 보면 두 런은 다음 차이가 있다.

- `054010`: resolve 단계 `threads=56`

## IPEX 실행 경로 분석

이번 H100x8 분석에서 중요한 점은 “CPU attention 이 IPEX 를 쓰는가”와 “IPEX 안에서도 어떤 함수가 실제 경로인가”를 분리해서 보는 것이다.

코드 기준 파일:

- `/vllm_hybrid/vllm/v1/attention/backends/cpu_attn.py`

### 1. 백엔드 선택

`_get_paged_attn_impl()` 는 IPEX import 가 가능하면 `_IPEXPagedAttention` 을 반환한다.

즉 H100x8 환경에서 IPEX 가 설치돼 있으면 decode attention 은 기본적으로 `_PagedAttention` fallback 이 아니라 `_IPEXPagedAttention` 경로를 탄다.

근거 코드:

```python
def _get_paged_attn_impl():
    if _is_ipex_available():
        return _IPEXPagedAttention
    else:
        return _PagedAttention
```

### 2. decode 시 실제 IPEX 함수

`_IPEXPagedAttention.forward_decode()` 에서 실제로 호출되는 함수는 다음이다.

```python
ipex_modules.PagedAttention.single_query_cached_kv_attention(
    output, query.contiguous(), key_cache, value_cache, head_mapping,
    scale, block_tables, context_lens, block_size, max_context_len,
    alibi_slopes)
```

의미:

- H100x8 hybrid decode 의 핵심 attention 연산은 IPEX 의
  `single_query_cached_kv_attention`
  으로 내려간다.
- 함수명 그대로 decode 는 “single-query + cached KV” 경로다.
- `context_lens.shape[0]` 가 batch 크기 역할을 하고, H1/H3 에서는 주로 `num_seqs=1`, H2 에서는 wave batch 로 인해 더 큰 batch 가 이 함수로 들어간다.

### 3. KV cache write 역시 IPEX 경로

`_IPEXPagedAttention.write_to_paged_cache()` 는 pure PyTorch cache write 가 아니라 IPEX 함수로 내려간다.

```python
ipex_modules.PagedAttention.reshape_and_cache(
    key, value, key_cache, value_cache,
    slot_mapping.flatten().int())
```

즉 H100x8 CPU decode 는

- KV cache write: `reshape_and_cache`
- KV cache read + decode attention: `single_query_cached_kv_attention`

의 조합으로 실행된다.

### 4. 이번 H100x8 실험에서 쓰이지 않은 IPEX 함수

`cpu_attn.py` 에는 prefix/chunked prefill 용 IPEX 함수도 있다.

```python
ipex_modules.PagedAttention.flash_attn_varlen_func(...)
```

하지만 이번 H100x8 hybrid 실험에서는 CPU 전용 config 생성 시

- `cpu_sched.enable_chunked_prefill = False`
- `cpu_sched.chunked_prefill_enabled = False`

로 강제 비활성화되어 있다.

코드:

- `/vllm_hybrid/vllm/v1/engine/hybrid_core.py`
- `_create_cpu_vllm_config()`

따라서 이번 실험에서 CPU prefill 은 “IPEX varlen flash attention 기반 chunked prefill”이 아니라, chunked prefill 이 꺼진 상태의 CPU 경로를 탄다. 즉 H100x8 결과를 해석할 때 IPEX 가 decode 에는 적극적으로 쓰였지만, CPU prefill 최적화 경로는 일부러 꺼져 있었다는 점을 분리해야 한다.

### 5. fallback 이 아니었다는 점

`_PagedAttention.forward_decode()` 는 CPU fallback 로직을 가진다.

- custom AVX-512 kernel
- batched SDPA
- loop-based SDPA

하지만 H100x8 환경에서는 `_get_paged_attn_impl()` 자체가 `_IPEXPagedAttention` 을 선택하므로, 정상 조건이라면 이 fallback 구현이 아니라 IPEX decode 함수가 호출된다.

즉 이번 H100x8 결과는 “Python SDPA fallback 이 느렸다”기보다 “IPEX decode 를 썼음에도 request-level hybrid tail 이 구조적으로 길었다”로 읽는 것이 맞다.

### 6. H2 (`max_seqs=16`) 재앙과 IPEX 의 관계

`max_seqs=16` 이라고 해서 decode 가 다른 attention backend 로 바뀌지는 않는다. 여전히 `_IPEXPagedAttention.forward_decode()` 가 호출되고, 결국 IPEX `single_query_cached_kv_attention()` 로 내려간다.

즉 H2 재앙의 1차 원인은:

- “IPEX 미사용”이 아니라
- “IPEX decode 를 사용하는 상태에서도 CPU engine 이 32 request tail 을 떠안은 것”

이다.

이 점은 중요하다. 현재 H100x8 문제는 “CPU 경로가 fallback 이라 느리다”보다 “CPU에 보내는 request 수 자체가 잘못 설계되면 아무리 IPEX 를 써도 wall time 이 무너진다”는 쪽에 가깝다.

### 7. 이번 분석으로 확정 가능한 IPEX 관련 결론

1. H100x8 hybrid decode attention 은 코드상 IPEX 경로를 사용한다.
2. 실제 decode attention 엔트리포인트는 `single_query_cached_kv_attention` 이다.
3. KV cache write 도 `reshape_and_cache` 로 IPEX 경로를 사용한다.
4. chunked prefill 기반 `flash_attn_varlen_func` 는 이번 실험에서는 CPU config 에서 비활성화되어 핵심 경로가 아니다.
5. 따라서 H100x8 tail 문제는 “IPEX 가 없어서 느림”이 아니라 “IPEX 를 써도 CPU가 GPU보다 너무 느리고, 그 CPU request 가 wall time 을 결정”하는 구조 문제다.

## wave-batch 구현이 로그에 어떻게 반영되는가

코드:

- `/vllm_hybrid/vllm/v1/engine/hybrid_core.py`

핵심 상태:

- `_cpu_wave_accepted`
- `_cpu_wave_closed`
- `_cpu_states[i]["in_flight"]`

동작:

1. CPU engine 이 admit 가능한 동안 request 수용
2. `accepted == cpu_max_num_seqs` 시 wave closed
3. closed 상태에선 추가 admit 없음
4. 해당 engine 의 `in_flight == 0` 이 되면 wave drained
5. drain 후 reset

로그와 연결:

- wave close:
  - `[HYBRID-WAVE] engine=X wave closed (accepted=N, batch_size=N)`
- wave drain:
  - `[HYBRID-WAVE] engine=X wave drained (accepted=N) → reset`

즉 이 마커는 라우터 내부 상태 전이를 직접 보여준다.

## `in_flight_cpu=2/1`, `32/16` 의 의미

이 수치는 혼동하기 쉽다.

- 분자: aggregate CPU inflight
- 분모: per-engine `cpu_max_num_seqs`

따라서:

- `2/1` with `num_cpu_engines=2` 는 정상 최대치
- `32/16` with `num_cpu_engines=2` 도 정상 최대치

overflow 버그를 뜻하지 않는다.

## 최종 결론

1. H100x8 hybrid 실험에서 CPU 실행 경로는 명백히 실제 동작한다.
2. NUMA-aware 2-engine 구조도 로그상 제대로 연결된다.
3. 현재 workload 에서는 CPU가 throughput 기여가 아니라 tail 만 만든다.
4. `cpu_max_num_seqs=16` 은 32개 CPU tail 을 만들어 재앙적이다.
5. `cpu_max_num_seqs=1` 은 그 tail 을 2개로 줄여 손실을 덜 키우는 방어적 설정이다.

실무적 해석:

- 이 결과는 “CPU 경로 bring-up 성공”의 증거다.
- 동시에 “현재 request-level hybrid 는 H100x8 7B burst 에서 구조적으로 이기기 어렵다”는 증거이기도 하다.

## RTX3090 결과와의 교차 확인

`eval/basic/RTX3090` 의 별도 결과도 같은 패턴을 보인다. 이 비교는 H100x8 로그 해석이 우연한 단발성 현상이 아니라 현재 hybrid 구조의 일반적 동작임을 뒷받침한다.

확인된 공통점:

- GPU bulk 가 먼저 거의 끝나고, 마지막 wall time 은 CPU tail 이 결정한다.
- `wave-batch` + `cpu-first` 조합이 메인 burst 초기에 CPU wave 를 먼저 채우고 닫는다.
- CPU 경로는 실제로 동작하지만, throughput 개선보다 tail penalty 를 만든다.

RTX3090 에서 확인된 구체 예:

- 1.5B hybrid:
  - `cpu_max_num_seqs=4`, `num_cpu_engines=1`
  - dispatch 결과 `cpu_count=4`, `gpu_count=496`
  - 이후 `GPU=497 reqs`, `CPU=4 reqs`, `in_flight_cpu=4/4`
  - 마지막 `wave drained (accepted=4)` 시점까지 wall time 이 늘어난다.
- 7B hybrid:
  - dispatch 결과 `cpu_count=4`, `gpu_count=96`
  - GPU bulk 완료 후에도 CPU 4 req 가 남아 약 1분 이상 tail 을 만든다.

이 비교로 보강되는 해석:

- H100x8 의 긴 tail 은 8x H100 특유의 이상 현상이라기보다 현재 request-level hybrid 정책 자체의 결과다.
- 환경이 달라도 CPU 가 맡은 request 는 GPU bulk 완료 이후 tail 로 남는 경향이 있다.
- H100x8 에서 `cpu_max_num_seqs=16` 이 재앙적인 이유는, RTX3090 에서도 이미 보이는 tail 구조가 더 큰 CPU wave 크기로 증폭되기 때문이다.

다음 분석 후보:

- H100x8 에서 왜 `112..223` 범위가 선택됐는지
- `threads=32`가 왜 최종 `56`으로 덮였는지
- CPU 1 request 당 대략 완료 시간과 token/s 추정
