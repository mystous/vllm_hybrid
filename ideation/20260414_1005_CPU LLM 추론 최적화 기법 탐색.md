# **극대규모 HPC 클러스터 환경에서의 CPU 기반 LLM 추론 및 베어메탈 수준 최적화 아키텍처**

## **서론: HPC 관점에서의 LLM 추론 병목 재정의**

기존의 일반적인 대규모 언어 모델(LLM) 서빙 아키텍처는 고수준(High-level)의 파이썬 래퍼(Wrapper)와 프레임워크에 의존하며, 단일 노드 또는 소규모 서버 환경의 한계에 갇혀 있다. 진정한 HPC(High-Performance Computing) 엔지니어링의 관점에서 볼 때, 수백억 개의 파라미터를 가진 모델을 수만 대의 CPU 코어가 엮인 클러스터에서 구동하기 위해서는 OS 스택과 프레임워크가 유발하는 오버헤드를 원천적으로 제거해야 한다.

본 보고서는 단순한 알고리즘적 우회술(예: 투기적 디코딩)이나 고수준 라이브러리 사용을 배제하고, **1\) 메모리 패킷의 직접 제어(RDMA/DMA), 2\) 실리콘 명령어 수준의 비트마스킹(AVX-512 Bitmasking)을 통한 희소성 극대화, 3\) JIT(Just-In-Time) 컴파일러를 통한 마이크로커널 튜닝, 4\) 정밀한 NUMA 토폴로지 통제와 하드웨어 캐시 프리패칭(Prefetching), 그리고 5\) C/C++ 기반의 베어메탈 코딩 아키텍처**라는 다섯 가지 핵심 HPC 엔지니어링 기법을 중심으로 CPU 클러스터의 LLM 추론 성능을 한계까지 끌어올리는 아키텍처를 심층 분석한다.

## **1\. RDMA 및 DMA 기반의 제로 카피(Zero-Copy) 분산 추론 아키텍처**

LLM의 디코딩(Decode) 단계에서 생성되는 방대한 KV 캐시(Key-Value Cache)를 수천 대의 노드 간에 동기화하는 작업은 클러스터 확장성의 가장 큰 적이다. 기존의 TCP/IP 소켓 통신이나 얕은 수준의 Message Passing 프로토콜은 커널 스페이스와 유저 스페이스 간의 반복적인 메모리 복사를 유발하여 치명적인 지연 시간(Latency)을 발생시킨다.

### **InfiniBand 및 RoCE를 활용한 OS 바이패스(Bypass)**

다중 노드 CPU 클러스터에서 KV 캐시를 교환하기 위해서는 운영체제의 개입을 완전히 차단하는 RDMA(Remote Direct Memory Access)가 필수적이다. InfiniBand나 RoCE(RDMA over Converged Ethernet) 프로토콜을 사용하면, 송신 노드의 시스템 메모리(DRAM)에서 수신 노드의 메모리로 데이터를 직접 쏴주는 제로 카피(Zero-copy) 데이터 전송이 가능해진다.

* **텐서 중심(Tensor-centric) 통신 매커니즘:** LLM의 KV 캐시는 연속적인 거대 메모리 청크가 아닌 다차원 텐서 조각으로 파편화되어 있다. 최적화된 HPC 클러스터에서는 RDMA 파이프라인 내부에 다중 슬롯 FIFO 버퍼를 구축하여, 디코딩 단계에서 토큰이 생성될 때마다 CPU 코어의 개입(Interrupts) 없이 네트워크 인터페이스 카드(NIC)의 DMA 엔진이 텐서를 직접 읽어다 목적지 노드로 비동기 전송한다.  
* **통신-연산 중첩(Overlap):** RDMA를 활용하면 CPU의 연산 유닛(ALU, AMX 등)은 어텐션 행렬 곱셈을 계속 수행하는 동안 백그라운드에서 KV 캐시의 분산 동기화가 이루어지므로, 통신으로 인한 파이프라인 스톨(Stall)을 거의 0에 가깝게 최소화할 수 있다.

## **2\. AVX-512 비트마스크(Bitmask) 명령어를 활용한 하드웨어 네이티브 희소성(Sparsity) 제어**

단순히 0이 많은 희소 행렬(Sparse Matrix)을 처리한다고 해서 성능이 올라가지 않는다. 일반적인 압축 포맷(CSR, CSC 등)이나 제로 패딩(Zero padding)은 벡터 레지스터 연산 시 막대한 메모리 대역폭 낭비와 분기 예측 실패(Branch misprediction)를 초래한다. 진정한 최적화는 ISA(Instruction Set Architecture) 레벨에서 이루어져야 한다.

### **비트마스크를 통한 무패딩(Zero-Padding Free) SpMV 커널**

최신 Intel 및 AMD CPU가 지원하는 AVX-512 명령어 셋 중, 마스크 레지스터(Mask Registers)를 직접 조작하는 K-명령어군(KMOVW, KMOVD, KMOVQ 등)을 극단적으로 활용해야 한다.

* **네이티브 비트마스킹(Native Bitmasking):** 가중치 행렬의 0이 아닌 값(Non-zero)의 위치를 메인 메모리에 정수형 배열로 저장하는 대신, 64비트 마스크 레지스터(k1\~k7)에 직접 로드한다. KMOVQ 명령어를 사용하면 64비트 크기의 희소 패턴을 한 번의 사이클로 벡터 유닛에 주입할 수 있다.  
* **희소 행렬-벡터 곱셈(SpMV) 극대화:** 이를 통해 메모리에서 패딩된 0 값을 아예 읽어오지 않고(Zero-padding free), 유효한 데이터만을 CPU의 L1/L2 캐시로 스트리밍하여 \_mm512\_mask\_fmadd\_ps와 같은 마스킹된 FMA(Fused Multiply-Add) 명령어로 즉각 연산한다. 이는 LLM 추론 시 50% 수준의 비정형 희소성(Unstructured Sparsity) 환경에서도 기존 Dense 연산 대비 1.5배 이상의 실질적인 속도 향상과 메모리 풋프린트 감소를 달성하는 핵심 기술이다.

## **3\. LibXSMM 및 JIT 컴파일 기반의 마이크로커널(Microkernel) 최적화**

Python 환경의 고수준 라이브러리나 벤더가 제공하는 범용 BLAS 라이브러리는 수만 가지의 엣지 케이스를 방어하기 위한 오버헤드로 가득 차 있다. 디코딩(Decode) 단계에서 발생하는 극도로 작고 좁은 형태(Tall and skinny)의 소규모 행렬 곱셈(Small-batch GEMM)은 이러한 범용 라이브러리를 통과할 때 캐시 미스와 스케줄링 병목에 갇히게 된다.

### **런타임 코드 생성(Runtime Code Generation)**

이를 극복하기 위해 LibXSMM과 같은 JIT(Just-In-Time) 컴파일 및 텐서 처리 원시 함수(Tensor Processing Primitives, TPPs) 인프라를 채택해야 한다.

* **동적 어셈블리 생성:** 어플리케이션 구동 시점에 모델의 파라미터 구조와 하드웨어 아키텍처(캐시 크기, AVX-512/AMX 유닛 유무)를 감지하여, 루프 오버헤드가 전혀 없는(Fully unrolled) 최적의 어셈블리 코드를 메모리에 직접 쏴서 실행한다.  
* 이러한 베어메탈 접근법은 메모리 대역폭 한계에 부딪히는 단일 배치(Single-batch) 어텐션 연산 시, 불필요한 메모리 로드/스토어를 제거하고 레지스터 블로킹(Register blocking)을 극대화하여 하드웨어의 이론적 대역폭 한계선(Roofline)에 완벽히 밀착된 성능을 이끌어낸다.

## **4\. NUMA 토폴로지 통제 및 L2/L3 캐시 정밀 프리패칭(Prefetching)**

멀티 소켓 랙(Rack) 서버 환경에서 OS의 기본 스케줄러(CFS)와 자동 NUMA 밸런싱(Auto NUMA balancing)에 메모리 관리를 맡기는 것은 HPC 엔지니어링의 직무 유기다. 소켓 간(Inter-socket) 메모리 엑세스는 로컬 메모리 접근 대비 지연 시간이 1.5배 이상 증가하며, QPI/UPI 링크 대역폭을 심각하게 소모한다.

### **메모리 바인딩 및 캐시 라인 제어**

LLM의 KV 캐시와 가중치 텐서는 철저히 하드웨어 인식적(Hardware-aware)으로 배치되어야 한다.

* **mbind와 mmap의 활용:** numactl 수준을 넘어, 추론 엔진의 메모리 할당기(Allocator) 단에서 OS 시스템 콜인 mmap과 mbind를 호출하여 특정 NUMA 노드의 물리적 뱅크에 페이지를 강제 핀(Pinning)해야 한다. CXL(Compute Express Link) 3.0 확장 메모리가 결합된 환경이라면, 페이지 인터리빙(Page-level interleaving)을 소프트웨어가 직접 통제하여 트래픽 편중을 막아야 한다.  
* **비동기 캐시 프리패칭(Asynchronous Cache Prefetching):** 메모리 버스에서 데이터가 도착하기를 기다리는 스톨(Stall)을 방지하기 위해, 소스 코드 내에 \_mm\_prefetch와 같은 인트린식(Intrinsic)을 의도적으로 삽입해야 한다. L2 및 L3 캐시 크기와 메모리 지연 시간에 맞춰 프리패치 거리(Prefetch Distance)를 계산하고(예: PREFETCH\_AHEAD 32), 연산이 활성화된 윈도우(Active computation window) 동안 후속 토큰 생성에 필요한 KV 캐시 라인을 L2 캐시로 비동기적으로 끌어올려야 한다. 이 기법을 정밀하게 적용하면 어텐션 커널의 하드웨어 효율을 2배 이상 끌어올릴 수 있다.

## **5\. C/C++ 기반 베어메탈(Bare-metal) 수준의 코딩 아키텍처 및 커널 융합**

파이썬과 같은 고수준 프레임워크의 오버헤드를 완전히 걷어내고, C/C++ 기반으로 메모리의 물리적 이동 경로와 연산 파이프라인을 극한으로 통제하는 코딩 최적화 기법은 다음과 같다.

* **사용자 공간(Userspace) DMA 및 NVMe 직접 제어 코딩:** 모델의 거대한 가중치를 로드할 때 운영체제의 페이지 캐시나 VFS(Virtual File System) 계층을 통과하면 엄청난 CPU 사이클이 낭비된다. 진정한 HPC C++ 추론 엔진은 OS를 우회하는 사용자 공간 NVMe 드라이버를 구축하여, 저장장치의 데이터를 CPU의 핀드 메모리(Pinned memory)로 DMA를 통해 직접 쏴주도록 코딩해야 한다. 연속된 플랫 버퍼(Flat-buffer)로 데이터를 수신하고 C++ 포인터 뷰(View)만 씌우는 제로 카피(Zero-Copy) 매커니즘을 적용하면, memcpy로 인한 지연 없이 즉각적인 포워드 패스(Forward pass) 연산이 가능하다.  
* **어셈블리 및 C++ 인트린식 레벨의 커널 융합(Kernel Fusion):** 메모리 장벽을 돌파하기 위해 여러 번의 메모리 왕복을 단일 루프로 병합해야 한다. RMSNorm, RoPE(회전 위치 임베딩), Attention 연산을 각각 독립적인 함수로 호출하는 대신, C++ 인트린식을 통해 이를 하나의 AVX-512 FMA(Fused Multiply-Add) 루프 안으로 강제 융합(Fusion)하도록 코딩하여 연산 사이클과 캐시 대역폭 낭비를 대폭 절약해야 한다.

## **결론**

엔터프라이즈 환경에서 환상으로 포장된 고수준 최적화 기법들은 수십, 수백 대의 노드까지만 통용될 뿐이다. 진정한 의미의 초거대 분산 CPU 추론 인프라를 구축하기 위해서는 개발자가 실리콘 레벨의 동작을 통제할 수 있어야 한다.

운영체제의 네트워크 스택을 우회하는 RDMA 제로 카피 파이프라인, AVX-512 비트마스크 명령어를 통한 제로 패딩 무효화, JIT 기반의 루프 언롤링 마이크로커널, 물리적 메모리와 캐시 라인의 완전한 통제, 그리고 OS를 배제한 순수 C/C++ 기반의 사용자 공간 DMA와 커널 융합 코딩만이 CPU 클러스터를 통한 LLM 연산의 한계를 부수고, 진정한 연산 밀도와 대역폭 확장을 이끌어낼 수 있는 유일하고 절대적인 방법이다.
