# 연산을 바꾸고 데이터 흐름을 재설계하라 — CPU LLM 추론의 근본 전환

**작성일**: 2026-04-13  
**관점**: 하드웨어 아키텍트 — "어떤 데이터를, 어떤 순서로 읽어서, 어떻게 계산하고, 어떻게 저장하느냐"  
**핵심 질문**: NPU/systolic array는 같은 행렬 곱셈을 왜 CPU보다 10~100배 빠르게 하는가? 그 원리를 CPU에 역수입할 수 있는가?

## 대전제: 문제는 연산량이 아니라 데이터 이동이다

7B BF16 디코드 1토큰의 실제 연산량:

```
FLOPs = 2 × 파라미터 수 = 2 × 7B = 14 GFLOPs
```

SPR AMX BF16 실효 처리량: ~30 TFLOPS. 14G / 30T = **0.47ms**. 이론상 디코드 1토큰이 **0.5ms 미만**에 끝나야 한다.

실측은 **~400ms**. 800배 차이. 연산은 0.5ms면 끝나는데 399.5ms가 데이터를 기다리는 시간이다. **연산이 아니라 데이터 이동이 99.9%의 시간을 차지한다.**

NPU와 systolic array가 빠른 이유는 연산이 빨라서가 아니다. **데이터를 한 번 읽으면 최대한 재사용하고, 읽는 순서를 제어하여 대기 시간을 0으로 만드는 데이터플로우 설계** 때문이다.

## 1. Systolic Array의 세 가지 데이터플로우 — CPU에서 배울 것

### 1.1 Weight Stationary (WS)

가중치를 PE(Processing Element)에 미리 로드하고 **고정**. 입력 데이터가 PE 배열을 흘러가며 가중치와 곱해진다.

```
PE[0][0]: w00 고정, 입력 x0 흘러옴 → w00 × x0 → 다음 PE로 전달
PE[0][1]: w01 고정, 입력 x0 흘러옴 → w01 × x0 → 다음 PE로 전달
...
```

**장점**: 가중치를 한 번만 로드. 입력이 여러 개면 같은 가중치를 재사용.  
**최적 조건**: **배치 추론** — 여러 입력이 같은 가중치를 공유할 때.

**CPU 매핑**: 가중치 타일을 L2 캐시에 로드하고 고정. 여러 입력 벡터(배치)가 순차적으로 같은 캐시된 가중치와 곱해진다. → **`cpu_max_num_seqs ≥ 2`의 하드웨어적 근거**. 배치=1에서는 WS의 이점이 0이고, 배치=4면 가중치 로드가 1/4로 줄어든다.

### 1.2 Output Stationary (OS)

출력 누적값을 PE에 고정. 입력과 가중치가 PE를 흘러가며, 부분합이 PE 내부 레지스터에 **누적**된다.

```
PE[i][j]: acc = 0
  for k in range(K):
    acc += A[i][k] × B[k][j]  // A, B가 흘러옴, acc는 PE에 머무름
output[i][j] = acc
```

**장점**: 부분합이 레지스터를 절대 떠나지 않음. 메모리 쓰기 최소화.  
**최적 조건**: **디코드(배치=1) GEMV** — 출력 벡터의 각 원소가 독립적으로 누적될 때.

**CPU 매핑**: AVX-512 FMA에서 `acc = _mm512_fmadd_ps(w, x, acc)`. `acc`는 zmm 레지스터에 상주하고, 가중치 `w`와 입력 `x`가 메모리에서 스트리밍된다. 현재 IPEX가 이미 이 패턴을 사용하지만, **핵심은 acc가 레지스터에서 최대한 오래 머물도록 루프를 설계하는 것**이다.

현재 문제: 서브레이어(QKV proj → O proj → gate → up → down)가 각각 독립 실행되면, 각 서브레이어마다 acc가 레지스터→메모리→레지스터를 왕복한다. **서브레이어를 융합**하면 중간 결과가 레지스터에 머물 수 있다.

### 1.3 Input Stationary (IS)

입력을 PE에 고정. 가중치가 PE를 흘러간다.

**CPU 매핑**: 입력 벡터 x(8KB)를 L1 캐시에 고정하고, 가중치 타일이 DDR에서 순차 스트리밍. 현재 GEMV의 자연스러운 패턴이 IS에 가깝다. 입력이 작고(8KB) 가중치가 크므로(14GB) 입력을 고정하는 것이 합리적이다.

### 1.4 LLM 디코드에 최적인 데이터플로우

| 단계 | 입력 크기 | 가중치 크기 | 출력 크기 | 최적 데이터플로우 |
|------|---------|-----------|---------|---------------|
| 디코드 BS=1 | 8 KB | 14 GB | 22 KB~88 KB | **IS + OS 하이브리드** |
| 디코드 BS=4 | 32 KB | 14 GB | 88~352 KB | **WS** (가중치 재사용) |
| 프리필 512tok | 4 MB | 14 GB | 4 MB | **WS** (가중치 재사용 극대화) |

**배치=1 디코드의 최적 전략**: 입력을 L1에 고정(IS), 출력 누적을 레지스터에 고정(OS), 가중치를 순차 스트리밍. 이것이 IS+OS 하이브리드다.

## 2. NPU의 3단 파이프라인 — CPU에 역수입

### 2.1 T-MAN의 DMA-Vector-Matrix 파이프라인

T-MAN(arXiv 2511.11248)은 Qualcomm Hexagon NPU에서 다음을 **동시에** 실행한다:

```
시간 →
[DMA  ] 타일 k+2 가중치를 DDR→TCM(온칩메모리)으로 전송
[Vector] 타일 k+1 가중치를 INT4→FP16로 LUT 역양자화
[Matrix] 타일 k   가중치 × 입력 = 부분합 (GEMM)
```

세 단계가 파이프라인되어 **DMA 레이턴시와 역양자화 오버헤드가 완전히 숨겨진다**. 결과: 프리필 1.4×, 디코드 3.1× 속도향상, 84% 에너지 절감.

### 2.2 CPU에서의 3단 파이프라인 매핑

SPR에서 동일한 3단 파이프라인을 구현할 수 있다:

```
시간 →
[DSA DMA     ] 타일 k+2 가중치를 DDR → L3 캐시로 전송
[AVX-512 코어] 타일 k+1 가중치를 INT4 → BF16 역양자화 (LUT 또는 shift-add)
[AMX TMUL    ] 타일 k   BF16 가중치 × 입력 = 부분합 (tdpbf16ps)
```

핵심 자원 매핑:

| NPU 자원 | CPU 대응 자원 | 특성 |
|---------|-----------|------|
| DMA 엔진 | **Intel DSA** (Data Streaming Accelerator) | 코어 독립, 비동기, 캐시 제어 가능 |
| Vector 코어 (HVX) | **AVX-512 FMA 유닛** | 역양자화, 원소별 연산 |
| Matrix 코어 (HMX) | **AMX TMUL** | 타일 행렬 곱셈 |
| TCM (온칩메모리, 8MB) | **L2 캐시 (코어당 2MB) + L3 (26MB/타일)** | 타일 스테이징 |

**NPU와의 차이점과 대응:**

- NPU의 TCM은 소프트웨어가 **직접 관리**하는 스크래치패드. CPU의 L2/L3은 하드웨어가 관리하는 캐시. → CPU에서는 `_mm_prefetch`와 DSA로 캐시 적재를 **간접 제어**
- NPU의 DMA는 프로그래머가 명시적으로 시작/종료. CPU의 DSA도 `enqcmd` 명령으로 명시적 제어 가능
- NPU의 Vector-Matrix 파이프라인은 하드웨어 인터록으로 동기화. CPU에서는 **코어 내 명령어 수준 병렬성(ILP)**으로 AMX 타일 연산과 AVX-512 역양자화를 겹침

### 2.3 구체적 구현 스케치

```c
// 3단 파이프라인 커널 (의사코드)
void gemv_3stage_pipeline(
    const int4_t* W_int4,  // INT4 가중치 (DDR)
    const bf16_t* x,       // 입력 벡터 (L1 상주)
    bf16_t* y,             // 출력 벡터
    int M, int K
) {
    // 타일 크기: 256×256 INT4 = 32KB (L2에 맞음)
    const int TILE_K = 256, TILE_M = 256;
    
    // 더블 버퍼: L2에 2개 타일 공간 확보
    bf16_t tile_buf[2][TILE_M * TILE_K];  // 각 128KB, L2에 상주
    
    // Stage 0: 첫 타일 프리페치 (DSA로 비동기)
    dsa_prefetch_to_l3(&W_int4[0], TILE_SIZE_INT4);
    
    // Stage 1: 첫 타일 역양자화 (AVX-512)
    dequant_int4_to_bf16_avx512(&W_int4[0], tile_buf[0], TILE_M, TILE_K);
    
    for (int t = 0; t < num_tiles; t++) {
        int cur = t % 2;
        int nxt = (t + 1) % 2;
        
        // [AMX] 현재 타일 GEMV 실행
        amx_gemv_bf16(tile_buf[cur], x, y, TILE_M, TILE_K);  // Stage 3
        
        if (t + 1 < num_tiles) {
            // [DSA] 다다음 타일 DDR→L3 프리페치 (비동기, 코어 블로킹 없음)
            if (t + 2 < num_tiles)
                dsa_prefetch_to_l3(&W_int4[(t+2) * tile_stride], TILE_SIZE_INT4);
            
            // [AVX-512] 다음 타일 역양자화 (AMX와 겹침)
            dequant_int4_to_bf16_avx512(&W_int4[(t+1) * tile_stride], 
                                         tile_buf[nxt], TILE_M, TILE_K);
        }
    }
}
```

## 3. LUT 기반 연산 대체 — 곱셈을 테이블 참조로 바꾸기

### 3.1 핵심 아이디어

INT4 가중치는 16개 값만 가질 수 있다(0~15 또는 -8~+7). 입력 활성화 x[i]에 대해 16개 가능한 곱 `x[i] × w` (w ∈ {0,...,15})를 **미리 계산**해서 테이블에 저장하면, 실제 곱셈이 **테이블 참조(vpshufb)**로 대체된다.

```
기존: load w → dequant w → FMA(x, w, acc)     [로드 + 변환 + 곱셈 + 덧셈]
LUT:  load w_idx → vpshufb(LUT, w_idx) → add   [로드 + 셔플 + 덧셈]
```

곱셈과 역양자화가 **모두 제거**되고, 테이블 참조(vpshufb, 1사이클)로 대체된다.

### 3.2 T-MAC: CPU에서 이미 동작하는 구현

T-MAC(Microsoft, EuroSys'25)은 정확히 이 방법을 x86 CPU에서 구현했다:

- INT1/2/3/4 가중치에 대해 LUT 기반 mpGEMM 제공
- AVX-512의 `vpshufb` (바이트 셔플) 명령어로 16-엔트리 LUT 참조
- **비트 수에 비례하는 선형 스케일링**: INT4 → INT2로 줄이면 속도 2× (기존 방법은 INT4→INT2에서 추가 속도향상 없음)
- Surface Laptop 7에서 llama.cpp 대비 **4× 속도향상**
- Llama-2-7B 4bit에서 NPU(10.4 tok/s)보다 CPU T-MAC(22 tok/s)이 **2배 빠름**

GitHub: https://github.com/microsoft/T-MAC

### 3.3 AVX-512에서의 LUT GEMV 구현

```c
// INT4 가중치 × FP16 활성화의 LUT 기반 GEMV
// 입력: x[K] (FP16), W_int4[M][K/2] (INT4 packed), y[M] (FP16)

void lut_gemv_int4(const uint8_t* W_int4, const __m512* x_fp16, 
                    float* y, int M, int K) {
    for (int m = 0; m < M; m++) {
        // 1. 이 출력 뉴런에 대한 16-엔트리 LUT 생성
        //    LUT[w] = Σ x[k] * dequant(w, scale, zero)  (w = 0..15)
        //    → x 값에 대해 16개 가능한 부분합을 미리 계산
        __m512 lut[16];
        for (int w = 0; w < 16; w++) {
            lut[w] = _mm512_set1_ps(dequant_value(w, scale[m], zero[m]));
        }
        
        // 2. K차원을 순회하며 LUT 참조로 부분합 누적
        __m512 acc = _mm512_setzero_ps();
        for (int k = 0; k < K; k += 32) {
            // INT4 가중치 16개 로드 (8바이트에 16개 INT4 packed)
            __m128i w_packed = _mm_loadu_si128((__m128i*)&W_int4[m * K/2 + k/2]);
            
            // 하위 4비트, 상위 4비트 분리 → LUT 인덱스
            __m128i lo = _mm_and_si128(w_packed, _mm_set1_epi8(0x0F));
            __m128i hi = _mm_srli_epi16(w_packed, 4);
            hi = _mm_and_si128(hi, _mm_set1_epi8(0x0F));
            
            // vpshufb로 LUT 참조 → 곱셈 결과를 직접 얻음
            __m128i result_lo = _mm_shuffle_epi8(lut_packed, lo);
            __m128i result_hi = _mm_shuffle_epi8(lut_packed, hi);
            
            // 누적
            acc = _mm512_add_ps(acc, expand_and_add(result_lo, result_hi, x_fp16[k/16]));
        }
        y[m] = _mm512_reduce_add_ps(acc);
    }
}
```

### 3.4 LUT의 CPU 캐시 적합성

| 양자화 비트 | LUT 크기 (FP16) | LUT 위치 | 참조 속도 |
|-----------|---------------|---------|---------|
| INT4 (16 엔트리) | 32 B | **L1 레지스터/캐시** | 1 사이클 |
| INT3 (8 엔트리) | 16 B | L1 레지스터 | 1 사이클 |
| INT2 (4 엔트리) | 8 B | 레지스터 | 1 사이클 |
| INT8 (256 엔트리) | 512 B | L1 캐시 | 4~5 사이클 |

INT4 LUT(32B)는 zmm 레지스터 하나에 들어간다. 즉 **LUT가 레지스터에 상주**하며 메모리 접근이 0이다. 이것이 LUT 방식이 기존 dequant+FMA보다 빠른 핵심 이유다.

## 4. 느린 연산을 빠른 연산으로 대체

### 4.1 역양자화(Dequantization) 제거

기존 INT4 GEMV:
```
[느림] INT4 로드 → INT8 확장 → FP16 변환 → 스케일 곱셈 → FP16 가중치
[느림] FP16 가중치 × FP16 입력 → FMA
```

LUT 기반:
```
[빠름] INT4 인덱스 로드 → vpshufb(LUT, 인덱스) → 부분합 직접 획득
```

역양자화의 **타입 변환(INT→FP) + 스케일 곱셈**이 완전히 제거된다. T-MAN은 이것으로 역양자화 커널 대비 **10.2× 속도향상**을 달성했다.

### 4.2 Softmax 대체

표준 Softmax:
```
[느림] exp(x_i) 계산 → 합산 → 나눗셈 → 정규화
```

`exp()` 함수는 x86에서 **~20 사이클** (AVX-512 `vexp2ps`는 근사값, 정확한 exp는 더 느림). 디코드 어텐션에서 매 토큰 × 매 헤드 × 매 시퀀스 위치마다 exp를 호출한다.

LUT 기반 Softmax:
```
[빠름] x_i를 LUT 인덱스로 양자화 → LUT[idx] = exp(x_i)의 사전계산값
→ vpshufb로 참조 → 합산 → 나눗셈
```

T-MAN에서 Softmax LUT로 **2.2× 속도향상**. CPU에서도 동일 적용 가능.

### 4.3 SiLU/GELU 활성화 함수 대체

SiLU(x) = x × sigmoid(x). sigmoid는 `exp(-x)` + 나눗셈을 포함.

**선형 근사 (TARDIS, arXiv 2501.10054)**:
- 입력의 80~90%가 좁은 범위에 집중
- 이 "핫 레인지"에서 SiLU(x) ≈ ax + b (선형 함수)로 근사
- 선형 함수면 Gate와 Up projection을 **사전 결합(constant folding)**할 수 있음
- 결과: FFN 가중치 **80% 감소**, vLLM에서 **1.6× 속도향상**

### 4.4 연산 대체 종합

| 원래 연산 | 사이클/원소 | 대체 연산 | 사이클/원소 | 배수 |
|----------|----------|---------|----------|------|
| INT4→FP16 역양자화 | ~8~12 | LUT vpshufb | ~1 | 8~12× |
| FP16 × FP16 FMA | ~4 | LUT 참조 + 정수 덧셈 | ~1~2 | 2~4× |
| exp() (Softmax) | ~20 | LUT 참조 | ~1 | ~20× |
| SiLU (sigmoid + mul) | ~25 | 선형 근사 (a×x+b) | ~1 | ~25× |
| RMSNorm (sqrt + div) | ~15 | rsqrt 근사 + FMA | ~5 | 3× |

## 5. 데이터 레이아웃 — 읽는 순서가 모든 것을 결정한다

### 5.1 가중치 패킹: 연산 순서에 맞춘 메모리 배치

**원칙**: 데이터가 메모리에 저장된 순서 = 연산이 데이터를 소비하는 순서. 이 둘이 일치하면 순차 접근이 되어 하드웨어 프리페처가 완벽하게 동작하고, 캐시라인 활용률이 100%가 된다.

**현재 PyTorch (행 우선)**:
```
메모리: W[0][0] W[0][1] ... W[0][N-1] | W[1][0] W[1][1] ...
GEMV y[j] = Σ W[i][j] × x[i]:  W[0][j], W[1][j], W[2][j], ... → 스트라이드 접근
```

**AMX 타일 최적 레이아웃**:
```
메모리: [타일(0:15, 0:31)] [타일(0:15, 32:63)] ... → tileloadd가 연속 16 캐시라인 로드
GEMV:  타일 순서대로 읽으며 계산 → 순차 접근
```

**INT4 LUT 최적 레이아웃 (T-MAC 스타일)**:
```
메모리: [그룹0의 4bit 가중치 128개] [그룹1의 4bit 가중치 128개] ...
       각 그룹이 하나의 LUT 스케일/제로와 대응
LUT GEMV: 그룹 순서대로 읽으며 LUT 참조 → 순차 접근
```

### 5.2 KV 캐시 레이아웃: 어텐션 접근 패턴에 맞추기

현재 vLLM:
```
KV cache: [block][token_in_block][head][dim]
어텐션: 하나의 헤드에 대해 모든 토큰의 K를 접근 → head가 내부 차원이라 스트라이드
```

최적:
```
KV cache: [head][block][token_in_block][dim]
어텐션: 하나의 헤드에 대해 모든 토큰의 K가 연속 → 순차 접근
```

### 5.3 활성화 레이아웃: 서브레이어 간 전달 최적화

트랜스포머의 서브레이어 체인:
```
QKV_proj → Attention → O_proj → Gate_proj → Up_proj → SiLU → Down_proj
```

각 서브레이어의 출력이 다음 서브레이어의 입력이 된다. 현재는 각 출력이 **메모리에 기록**된 후 다음 서브레이어가 다시 읽는다. 서브레이어를 융합하면 중간 결과가 **레지스터/L1에 머물** 수 있다.

## 6. 코어 간 데이터 흐름 — systolic 원리의 멀티코어 적용

### 6.1 현재: 각 코어가 독립적으로 DDR을 읽음

```
코어 0: DDR → [자기 슬라이스의 가중치] → 연산 → DDR에 부분합 저장
코어 1: DDR → [자기 슬라이스의 가중치] → 연산 → DDR에 부분합 저장
...
코어 47: DDR → [자기 슬라이스의 가중치] → 연산 → DDR에 부분합 저장
최종: 48개 부분합을 메모리에서 합산 (reduce)
```

코어 간 데이터 공유가 **0**이다. 각 코어가 독립적인 프로세서처럼 동작한다.

### 6.2 systolic 스타일: 코어 간 데이터 전달

systolic array에서 데이터는 PE 간에 **직접 전달**된다. CPU 코어 간에도 L3 캐시를 통해 데이터를 전달할 수 있다:

```
[파이프라인 분할: 레이어를 코어 그룹에 배정]

코어 그룹 A (0~11):  서브레이어 1~3 (QKV proj)
  → 결과를 L3 공유 영역에 기록

코어 그룹 B (12~23): 서브레이어 4 (Attention)
  → L3에서 QKV 결과를 읽고, O proj 결과를 L3에 기록

코어 그룹 C (24~35): 서브레이어 5~7 (MLP: Gate + Up + SiLU + Down)
  → L3에서 O proj 결과를 읽고, 최종 출력을 L3에 기록

코어 그룹 D (36~47): 다음 레이어의 서브레이어 1~3
  → 파이프라인: 현재 레이어의 MLP가 끝나기 전에 다음 레이어 QKV 시작
```

이 구조에서 코어 그룹 간 데이터 전달은 **L3 캐시를 경유**하며, L3 대역폭(~300 GB/s)이 DDR 대역폭(~250 GB/s)보다 높으므로 병목이 되지 않는다. 그리고 각 코어 그룹은 자기 서브레이어의 가중치만 L2에 보관하면 되므로, **L2 캐시에 서브레이어 가중치가 맞는다** (7B QKV proj = 3 × 4096 × 4096 × 2B = 96MB → 12코어 × 2MB L2 = 24MB에 타일 단위로 순환 적재).

### 6.3 프로듀서-컨슈머 파이프라인

```
시간 →

그룹A: [레이어0 QKV] [레이어1 QKV] [레이어2 QKV] ...
그룹B:            [레이어0 Attn] [레이어1 Attn] [레이어2 Attn] ...
그룹C:                        [레이어0 MLP] [레이어1 MLP] ...
그룹D:                                    [레이어0 다음] ...
```

각 그룹이 이전 그룹의 결과가 나오자마자 처리를 시작한다. **4개 레이어가 동시에 파이프라인 실행**되므로 코어 활용률이 극대화된다.

## 7. 연산 강도를 높이는 수학적 변환

### 7.1 Gate + Up 연산 결합

SwiGLU MLP:
```
gate = W_gate × x      (GEMV 1)
up   = W_up × x        (GEMV 2)  
out  = SiLU(gate) * up  (원소별)
down = W_down × out     (GEMV 3)
```

Gate와 Up이 **같은 입력 x**를 사용한다. 두 가중치를 인터리브하면:

```
W_combined = interleave(W_gate, W_up)  // [M, 2K] → [M, K]의 2배 열
combined = W_combined × x  // 단일 GEMV
gate = combined[::2]  // 짝수 열
up = combined[1::2]   // 홀수 열
```

입력 x를 **한 번만 로드**하고 두 GEMV를 동시에 처리한다. 메모리 접근 절반 감소, 산술 강도 2배 증가.

### 7.2 QKV Projection 결합

QKV도 같은 입력에 대해 세 번 GEMV를 한다:

```
Q = W_q × x,  K = W_k × x,  V = W_v × x
```

`W_qkv = concat(W_q, W_k, W_v)` (행 방향 결합) 로 단일 GEMV:

```
QKV = W_qkv × x  // 한 번의 GEMV로 Q, K, V 동시 생성
```

GQA(8 KV 헤드)에서 W_q(4096×4096) + W_k(4096×1024) + W_v(4096×1024) = W_qkv(4096×6144). 입력 x 로드가 3회 → 1회로 감소.

### 7.3 저랭크 분해

W ∈ R^{4096×11008}를 SVD로 W ≈ U × S × V^T, rank r=256:

```
원래: y = W × x         → 4096 × 11008 × 2 FLOPs, 86MB 메모리 접근
저랭크: t = V^T × x      → 256 × 11008 × 2 FLOPs, 5.5MB
        y = U × (S × t)  → 4096 × 256 × 2 FLOPs, 2MB
합계: 7.5MB 메모리 접근 (원래의 9%), 연산 86% 감소
```

정확도 손실이 있지만, 투기적 디코딩의 **드래프트 경로**에 사용하면 (정확한 결과가 불필요) 극적인 가속이 가능하다.

## 8. 종합: 데이터 흐름 재설계 로드맵

| 순서 | 기법 | 원리 | 예상 효과 |
|------|------|------|---------|
| 1 | **LUT 기반 GEMV (T-MAC)** | 곱셈→테이블 참조 | INT4에서 **2~4×** |
| 2 | **가중치 블록 인터리브 패킹** | 순차 접근 = 캐시라인 100% | **2~3×** |
| 3 | **Gate+Up 인터리브 결합** | x 로드 1회로 2 GEMV | **1.5~1.8×** |
| 4 | **QKV 결합** | x 로드 1회로 3 GEMV | **1.2~1.3×** |
| 5 | **3단 파이프라인 (DSA+AVX+AMX)** | 로드/역양자화/연산 겹침 | **1.5~2×** |
| 6 | **코어 그룹 파이프라인** | 서브레이어 병렬 실행 | **2~3×** (레이턴시) |
| 7 | **Softmax/SiLU LUT 대체** | 초월함수→테이블 참조 | **1.1~1.2×** |
| 8 | **OS 데이터플로우 (레지스터 누적)** | 중간 결과 메모리 왕복 제거 | **1.2~1.3×** |

**1~5를 모두 적용하면 이론적 누적 효과: ×9~26**. 2.3 tok/s → **~20~60 tok/s** (7B). 이전 문서의 TLB/페이지/프리페치 최적화와 결합하면 이론 한계(INT4 기준 71 tok/s)에 근접 가능하다.

## 핵심 메시지

NPU가 빠른 이유는 특별한 트랜지스터가 아니다. **데이터를 읽는 순서, 계산하는 방법, 저장하는 위치를 하드웨어 특성에 맞춰 최적화**했기 때문이다. CPU에도 DSA(DMA), AMX(행렬 연산), AVX-512(벡터 연산), L1/L2/L3(다단 캐시)가 있다. 이 자원을 NPU처럼 **3단 파이프라인으로 조율하고, LUT로 느린 연산을 제거하고, systolic 원리로 코어 간 데이터를 흘리면**, CPU도 메모리 대역폭 한계까지 성능을 끌어올릴 수 있다.

파이썬 레벨에서 `model.forward()`를 호출하는 것은 이 모든 것의 위에 앉아 있는 얇은 막이다. 진짜 성능은 그 아래, 캐시라인과 타일 레지스터와 DDR 뱅크가 만나는 곳에서 결정된다.
