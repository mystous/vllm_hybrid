\section{배경}
\label{sec:background}

\subsection{투기적 디코딩의 동작 원리}
\label{subsec:spec-basics}

autoregressive 디코딩은 토큰을 \emph{한 개씩} 생성하므로,
매 토큰마다 모델 전체 가중치를 메모리에서 읽어야 하는
memory-bandwidth-bound 연산이 된다.
이 과정에서 GPU 연산 유닛은 대부분 유휴 상태로 남는다.

투기적 디코딩~\cite{leviathan2023spec,chen2023spec}은
\emph{draft-then-verify} 구조로 이 비효율을 줄인다.

\textbf{무엇을 예측하는가.}
드래프터(drafter)는 \emph{타깃 모델이 다음에 생성할 토큰들}을 미리 추측한다.
새로운 출력을 만드는 것이 아니라, 큰 모델(타깃)이 내놓을 결과를
앞질러 흉내내는 것이다.
드래프터가 제안한 $K$개의 후보 토큰을 타깃 모델이 \emph{단일 forward pass}로
한 번에 검증하여, 일치하는 접두부(prefix)를 수락하고
처음 불일치하는 토큰부터 타깃 모델 값으로 교정한다.

\textbf{왜 빨라지는가.}
일반 디코딩이 토큰 $K$개에 forward $K$번을 쓰는 반면,
투기적 디코딩은 드래프터가 맞힌 만큼을 forward \emph{한 번}으로 진행한다.
드래프터 호출은 타깃 모델 forward보다 훨씬 저렴하므로,
추측이 자주 맞으면 전체 처리량이 향상된다.
예를 들어 타깃 모델의 실제 출력이 \texttt{"the cat sat on the"}일 때
드래프터가 \texttt{"the cat sat on a"}를 제안하면,
앞 4개(\texttt{the cat sat on})가 수락되고 마지막만 교정되어
forward 1번으로 5토큰이 확정된다(일반 디코딩은 forward 5번 필요).

\textbf{품질 보존.}
드래프터가 틀려도 최종 출력은 변하지 않는다.
불일치 토큰은 rejection sampling~\cite{leviathan2023spec}으로
타깃 모델 분포에서 다시 샘플링되어, 출력이 타깃 모델 단독 디코딩과
동일함이 수학적으로 보장된다(lossless).
즉 드래프터의 역할은 \emph{정확도가 아니라 속도}이며,
잘 맞히면 빨라지고 못 맞히면 (검증 오버헤드만큼) 느려질 뿐이다.
이 손익을 정량화한 것이 다음의 R/K 프레임워크이다.

\subsection{R/K 프레임워크}
\label{subsec:rk-framework}

투기적 디코딩의 처리량 이득은 R/K 비율로 정량화된다~\cite{leviathan2023spec}.
$K$개의 후보 토큰 중 평균 수락 토큰 수는

\begin{equation}
\mathbb{E}[\text{accepted}] = \frac{1 - \alpha^{K+1}}{1 - \alpha}
\label{eq:expected-accept}
\end{equation}

이며, 여기서 $\alpha \in [0,1]$은 토큰 단위 수락률이다.
스텝당 오버헤드 비율 $R$과 평균 수락 토큰 수 $K$의 관계로
순이익 조건이 결정된다.

\begin{equation}
\frac{T_{\text{spec}}}{T_{\text{base}}} \approx \frac{R}{K}, \quad
K > R \Rightarrow \text{gain}, \quad K < R \Rightarrow \text{regression}
\label{eq:rk-ratio}
\end{equation}

$R$은 모델 크기와 TP 구성에 의존하며, $K$는 드래프터 품질과 워크로드 특성에 의존한다.
소형 모델($\leq$7B)에서 $R$이 커지는 이유는
스펙 디코딩 검증 forward pass의 상대 비용이 증가하기 때문이다.

\textbf{$R$ 산출 방법.}
$R$은 이론값이 아니라 실측값으로, 다음과 같이 정의된다.
\begin{equation}
R = \frac{T_{\text{spec\_step}}}{T_{\text{base\_step}}}
\label{eq:r-measured}
\end{equation}
투기적 디코딩 스텝은 드래프트(draft) 생성과 타깃 모델 검증(verify)의
두 단계를 포함하므로, 동일 모델의 baseline 스텝보다 오버헤드가 발생한다.
따라서 $R > 1$이 일반적이며, 이 값은 하드웨어 구성과 TP 설정에 따라 달라진다.
본 연구의 측정 환경(Qwen-2.5-32B + TP=4$\times$2 + H100$\times$8)에서
$R \approx 1.3$이며, $K < 1.3$인 드래프터 설정은 net-negative가 된다.

\textbf{단일 요청 모델의 한계.}
식~\ref{eq:rk-ratio}는 \emph{단일 요청}의 디코딩 지연 모델이다.
반면 \S\ref{sec:results}의 처리량(tps) 측정은 continuous batching~\cite{yu2022orca}
하에서 이루어지므로, 배치 내 요청 간 자원 공유와 스케줄링 효과가 더해진다.
따라서 R/K 모델은 ``$K > R$이면 가속, $K < R$이면 회귀''라는
\emph{부호와 방향}에 대한 직관을 제공하는 정성적 도구로 사용하며,
처리량 향상의 정확한 크기는 식에서 직접 도출하지 않고 실측으로 보고한다.

\subsection{vLLM의 투기적 디코딩 구현}
\label{subsec:vllm-spec}

vLLM v1~\cite{kwon2023vllm}은 투기적 디코딩을
\emph{드래프터에 독립적인}(drafter-agnostic) 프레임워크로 구현한다.
핵심은 두 컴포넌트로 분리된다.

\textbf{Proposer (드래프터).}
다음 후보 토큰들을 생성하는 모듈로, 플러그인 형태로 교체 가능하다.
n-gram(프롬프트 내 패턴 매칭), SuffixDecoding(suffix tree),
EAGLE(경량 헤드) 등이 동일 인터페이스로 끼워진다.
\S\ref{subsec:spec-basics}의 ``예측''이 여기서 일어난다.

\textbf{Rejection sampler (검증·교정).}
타깃 모델이 후보 위치들을 단일 forward로 통과시켜 logits를 구하면,
rejection sampler가 수락/거부를 결정하고 거부 지점부터 재샘플링한다.
이 단계가 출력의 lossless 보존을 담당한다.
거부된 후보 토큰의 KV 캐시 슬롯은 PagedAttention~\cite{kwon2023vllm}이
회수하여 메모리 낭비를 막는다.

이 분리 덕분에 본 연구는 vLLM 코어를 거의 수정하지 않고
서로 다른 드래프터(n-gram, suffix)를 동일 조건에서 공정 비교할 수 있다.

\subsection{SuffixDecoding}
\label{subsec:suffix-decoding}

\textbf{무엇으로 예측하는가.}
n-gram 드래프터가 \emph{입력 프롬프트 내부}의 반복 패턴만 보는 데 반해,
SuffixDecoding~\cite{snowflake-arctic}은 입력 프롬프트와
\emph{이전에 생성된 출력}을 모두 suffix tree에 누적한다.
새 토큰을 예측할 때는 현재까지의 문맥 suffix를 트리에서 매칭하여,
과거에 그 패턴 뒤에 등장했던 토큰들을 빈도 가중치로 후보로 제안한다.
즉 ``방금 본 패턴이 전에도 나왔다면 그 다음도 같을 것''이라는
self-similarity를 활용한다.

\textbf{Adaptive draft length.}
SuffixDecoding은 최대 $K_{\max}$개까지 후보를 내되,
트리 매칭(hit)이 강할수록 길게, 약하거나 없으면 draft를 $1$개로 줄인다
($1 \leq K \leq K_{\max}$).
n-gram의 고정 길이 draft와 달리, hit가 없을 때 짧게 끊어
검증 오버헤드 $R$을 최소화하는 것이 핵심 강점이다.

이 두 성질이 결합되어 code 워크로드에서 큰 차이를 만든다.
코드는 함수 정의, 반복 호출, 들여쓰기 등 self-similar 패턴이 많아
이전 생성 출력이 suffix tree에 쌓일수록 $\alpha$가 향상된다
($\alpha_{\text{code}}^{\text{ngram}} = 1.2\%$ vs.\ $\alpha_{\text{code}}^{\text{suffix}} = 40.1\%$,
$K$ 비율 $3.74\times$, $\alpha$ 비율 $33\times$).

\subsection{CUDA Graph와 PIECEWISE 모드}
\label{subsec:piecewise}

\textbf{CUDA Graph가 필요한 이유.}
디코딩 스텝마다 수백 개의 작은 CUDA 커널이 실행되는데,
각 커널을 개별 launch하면 CPU 측 launch 오버헤드가 누적되어
GPU가 굶는다(\S\ref{subsec:spec-basics}의 feeding 문제와 동일 맥락).
CUDA Graph는 이 커널 시퀀스를 \emph{한 번 캡처}한 뒤
\emph{replay}만 하여 launch 오버헤드를 제거한다.

\textbf{FULL vs PIECEWISE.}
FULL 모드는 forward 전체를 하나의 그래프로 캡처하므로
입출력 shape가 \emph{고정}되어야 한다.
그러나 투기적 디코딩은 스텝마다 수락 토큰 수($K$)가 달라
입력 shape가 \emph{동적}으로 변한다.
따라서 SuffixDecoding의 adaptive draft length는 FULL 모드와 비호환이다.
vLLM v1의 \texttt{cudagraph\_mode=PIECEWISE}는
shape가 고정된 부분 그래프(예: layer 내부 연산)만 캡처하고,
동적 shape를 다루는 연산은 그래프 밖에 두어
투기적 디코딩과 CUDA Graph 가속을 동시에 얻는다.
본 연구의 spec-decode engine은 이 PIECEWISE 모드를 기본으로 사용한다.

\subsection{듀얼-NUMA 하드웨어 아키텍처}
\label{subsec:numa-arch}

본 연구의 측정 환경은 듀얼 NUMA 노드 Xeon SPR-class 서버다.
각 NUMA 노드는 56개 물리 코어와 약 1\,TB DRAM을 보유하며,
remote/local 접근 거리 비는 $2.1\times$(local 10, remote 21)이다.

GPU PCIe 연결도 NUMA-aware하다:
GPU 0--3은 NUMA node 0에, GPU 4--7은 NUMA node 1에
PCIe affinity가 있다.
이중 백엔드에서 baseline 인스턴스(GPU 0--3)를 NUMA node 0에,
spec-decode 인스턴스(GPU 4--7)를 NUMA node 1에 바인딩하면
cross-socket QPI 트래픽이 제거된다.

\subsection{vLLM 아키텍처}
\label{subsec:vllm-arch}

vLLM~\cite{kwon2023vllm}은 PagedAttention 기반 KV 캐시 가상화와
continuous batching~\cite{yu2022orca}으로 GPU 메모리 효율과 처리량을 최적화한다.
본 연구는 vLLM v1.6.dev0+g858b6df7a 포크를 기반으로 하며,
vLLM 코어 변경은 하위 호환성을 완전히 유지하는 14줄에 불과하다:
\texttt{vllm/utils/\_\_init\_\_.py} ($+5$ 줄, \texttt{FlexibleArgumentParser} re-export)와
\texttt{vllm/engine/arg\_utils.py} ($+9$ 줄, \texttt{\_is\_v1\_supported\_oracle} stub).
