# TSK_004 — Cold-KV 경로 NUMA-aware 화

| 항목 | 값 |
|---|---|
| ID | `TSK_004` |
| 상태 | `활성` (Phase 1 dev 구현 완료. Phase 2 prod 측정은 [`TST_002`](TST_002.md) 와 함께 사용자 직접 진행) |

| 검증 게이트 | [`TST_002`](TST_002.md) — 별도 TST 신설 안 함, throughput / overlap profile 게이트가 NUMA 페널티 제거 효과까지 흡수 |
| 매핑 IDE_006 진입 조건 | (b) throughput net-win 의 신뢰성 확보 (NUMA 페널티가 측정값을 오염시키지 않도록) |

> **왜 분리해서 따로 추적하는가**: dev 머신 (단일 socket Core i9-12900KF) 은 NUMA 가 단일 노드라 무관하지만, prod 타깃 (dual-socket Sapphire Rapids 8480+) 은 노드가 두 개로 나뉘어 한쪽 socket worker 가 반대편 socket 의 CPU KV 페이지를 읽으면 cross-socket access latency 페널티 (~2~3×) 가 발생합니다. 정확도 (TST_003) 에는 영향 없으나 [`TST_002`](TST_002.md) 의 throughput / overlap profile 측정값을 직접 왜곡하므로, throughput 단계 진입 전에 처리해야 net-win 판정이 NUMA 노이즈에 오염되지 않습니다.

## 1. 목적과 범위

### 1.1 · 목적

OffloadingConnector 의 CPU staging buffer 와 [`TSK_001`](TSK_001.md) / [`TSK_003`](TSK_003.md) 의 CPU partial-attention 커널이 **자기 worker 의 NUMA node 에서 메모리 접근과 연산을 모두 수행** 하도록 만든다. 결과적으로 모든 cold KV read 가 local memory access 로 떨어지고 cross-socket UPI 트래픽이 사라져 [`TST_002`](TST_002.md) 의 측정값이 알고리즘 자체의 비용만 반영하게 된다.

### 1.2 · 범위

| 단계 | 내용 | 진행 상태 |
|---|---|---|
| (a) connector buffer NUMA bind | `vllm/distributed/kv_transfer/kv_connector/v1/offloading/numa_aware.py:bind_worker_to_local_numa()` 가 `vllm/v1/kv_offload/worker/cpu_gpu.py:CpuGpuOffloadingHandlers.__init__` 의 `torch.zeros(..., pin_memory=True)` 직전에 호출. `vllm.platforms.intel_cpu_utils.NUMAAllocator.bind_to_node` 가 내부적으로 libnuma 의 `numa_set_preferred(node)` 를 호출해 이 thread 의 후속 alloc 이 자기 socket 을 선호. local node 결정: `current_platform.get_device_numa_node(local_rank)` (worker 의 GPU NUMA node) 우선 → `rank % num_numa_nodes` fallback → topology 미감지 시 skip. libnuma 미설치 / 단일 socket / 비-Linux 시 silent no-op | **구현 완료** (Phase 1 dev) — dev 단일 socket 에서 정상 idempotent 호출 확인. prod 효과 측정은 (d) 에서 |
| (b) partial-attention 커널 thread bind | `numa_aware.py:pin_threads_to_local_numa()` 가 `vllm/v1/attention/ops/cpu_partial_attention.py:forward_partial_with_lse` 첫 진입 시 `os.sched_setaffinity(0, cpus_of_local_node)` 로 worker 프로세스 affinity 를 자기 NUMA node 의 코어에 묶음. OpenMP / std::thread 가 그 affinity 마스크를 inherit 하므로 별도 OMP_PROC_BIND / GOMP_CPU_AFFINITY 설정 없이 효과 발생. idempotent — `_pin_done` 캐시로 중복 호출 비용 0 | **구현 완료** (Phase 1 dev) — 단위 호출 시 `pinned threads to NUMA node N (M cores)` 로그 발화 확인 |
| (c) 큰 cpu_bytes_to_use 의 분할 | 본 단계는 (a) 의 numa_set_preferred 가 worker 별로 다른 node 를 선호하게 만들면 자연스럽게 해결. 명시적 분할 없이 8 worker 의 first-touch 가 4+4 로 흩어지면 사실상 두 socket 에 균등 배치. 추가 분할이 필요할 만큼 worker 간 메모리 압박이 비대칭으로 떨어지는 케이스가 있으면 후속으로 검토 | **현 단계 불필요** — (a) 의 per-worker preferred node 만으로 충분 |
| (d) 검증 | [`TST_002`](TST_002.md) sweep 결과의 socket-local vs cross-socket 비교. `numactl --hardware` + `numastat -p <pid>` 로 페이지 분포 확인 | **prod 측정 대기** (사용자 직접) |

### 1.3 · 비범위

- 정확도 (TST_003 — D-i / D-ii) — NUMA 무관, 본 TSK 와 별개
- multi-host 분산 KV — IDE_006 의 단일 호스트 가정 안에서만 적용
- GPU↔CPU 전송 자체의 NUMA-aware 화 (`cudaHostRegister` 의 NUMA 힌트) — 별도 후속 검토. 본 TSK 는 CPU-side 만 다룸
