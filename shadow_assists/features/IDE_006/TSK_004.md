# TSK_004 — Cold-KV 경로 NUMA-aware 화

| 항목 | 값 |
|---|---|
| ID | `TSK_004` |
| 상태 | `활성` (Phase 1 dev 구현 완료. Phase 2 prod 측정은 [`TST_002`](TST_002.md) 와 함께 사용자 직접 진행) |
| 부모 PLN | [`PLN_001`](PLN_001.md) |
| 조부 IDE | [`IDE_006`](README.md) |
| 선행 TSK | [`TSK_001`](TSK_001.md), [`TSK_002`](TSK_002.md), [`TSK_003`](TSK_003.md) (모두 NUMA 무인지 상태로 일단 동작 — 본 TSK 가 그 위에 NUMA 인지 레이어를 얹음) |
| 검증 게이트 | [`TST_002`](TST_002.md) — 별도 TST 신설 안 함, throughput / overlap profile 게이트가 NUMA 페널티 제거 효과까지 흡수 |
| 매핑 IDE_006 진입 조건 | (b) throughput net-win 의 신뢰성 확보 (NUMA 페널티가 측정값을 오염시키지 않도록) |
| ID 넘버링 출처 | [`shadow_assists/id_registry.md`](../../id_registry.md) |

> **왜 분리해서 따로 추적하는가**: dev 머신 (단일 socket Core i9-12900KF) 은 NUMA 가 단일 노드라 무관하지만, prod 타깃 (dual-socket Sapphire Rapids 8480+) 은 노드가 두 개로 나뉘어 한쪽 socket worker 가 반대편 socket 의 CPU KV 페이지를 읽으면 cross-socket access latency 페널티 (~2~3×) 가 발생합니다. 정확도 (TST_003) 에는 영향 없으나 [`TST_002`](TST_002.md) 의 throughput / overlap profile 측정값을 직접 왜곡하므로, throughput 단계 진입 전에 처리해야 net-win 판정이 NUMA 노이즈에 오염되지 않습니다.

## 1. 목적과 범위

### 1.1 · 목적

OffloadingConnector 의 CPU staging buffer 와 [`TSK_001`](TSK_001.md) / [`TSK_003`](TSK_003.md) 의 CPU partial-attention 커널이 **자기 worker 의 NUMA node 에서 메모리 접근과 연산을 모두 수행** 하도록 만든다. 결과적으로 모든 cold KV read 가 local memory access 로 떨어지고 cross-socket UPI 트래픽이 사라져 [`TST_002`](TST_002.md) 의 측정값이 알고리즘 자체의 비용만 반영하게 된다.

### 1.2 · 범위

| 단계 | 내용 | 진행 상태 |
|---|---|---|
| (a) connector buffer NUMA bind | `vllm/distributed/kv_transfer/kv_connector/v1/offloading/numa_aware.py:bind_worker_to_local_numa()` 가 `vllm/v1/kv_offload/worker/cpu_gpu.py:CpuGpuOffloadingHandlers.__init__` 의 `torch.zeros(..., pin_memory=True)` 직전에 호출. `vllm.platforms.intel_cpu_utils.NUMAAllocator.bind_to_node` 가 내부적으로 libnuma 의 `numa_set_preferred(node)` 를 호출해 이 thread 의 후속 alloc 이 자기 socket 을 선호. local node 결정: `current_platform.get_device_numa_node(local_rank)` (worker 의 GPU NUMA node) 우선 → `rank % num_numa_nodes` fallback → topology 미감지 시 skip. libnuma 미설치 / 단일 socket / 비-Linux 시 silent no-op | **구현 완료** (Phase 1 dev + prod) — dev 단일 socket idempotent 호출 확인. prod log 에서 `rank=N bound CPU memory preference to NUMA node M` × 8 worker. |
| (b) partial-attention 커널 thread bind | `numa_aware.py:pin_threads_to_local_numa()` 가 `vllm/v1/attention/ops/cpu_partial_attention.py:forward_partial_with_lse` 첫 진입 시 `os.sched_setaffinity(0, cpus_of_local_node)` 로 worker 프로세스 affinity 를 자기 NUMA node 의 코어에 묶음. OpenMP / std::thread 가 그 affinity 마스크를 inherit 하므로 별도 OMP_PROC_BIND / GOMP_CPU_AFFINITY 설정 없이 효과 발생. idempotent — `_pin_done` 캐시 + `_pin_fast_done` 모듈-level bool 로 첫 성공 후 hot path 의 lock 진입까지 단축 | **구현 완료** (Phase 1 dev + prod) — 단위 호출 시 `pinned threads to NUMA node N (M cores)` 로그 발화 확인 |
| (b') worker 별 NUMA node cpulist **분할** | 한 NUMA node 에 여러 TP worker 가 같이 묶이는 경우 (prod TP=8 / 2 socket → 4 workers per node), worker 마다 노드 전체 cpulist 를 그대로 affinity 로 박으면 4 × 56 = 224 thread 를 56 코어에 spawn 시키는 storm 이 발생해 `libgomp pthread_create` 가 EAGAIN. `numa_aware.py:_partition_node_cpus_for_rank()` 가 같은 node 에 묶인 rank 들을 round-robin 으로 식별해 노드 cpulist 를 worker 수만큼 균등 분할 (잉여 코어는 앞 슬라이스부터 1 개씩 우선 배분), `pin_threads_to_local_numa()` 가 그 슬라이스를 affinity 로 사용. 이렇게 되면 C++ helper `vllm_partial_attn_thread_count()` 가 사용하는 `sched_getaffinity` 결과가 자동으로 worker 당 정당한 몫 (예: prod 56/4 = 14 코어, SMT 포함 28/112) 을 반환 | **구현 완료** (Phase 1 dev simulation + Phase 2 prod 검증) — prod log: `rank=N pinned threads to NUMA node M (28/112 cores: A~B)` × 8 worker. 이전 EAGAIN 회귀 사라짐 |
| (b'') C++ helper 의 OMP `num_threads()` 강제 | vLLM V1 multiproc executor 가 worker subprocess 에 `OMP_NUM_THREADS=1` 을 강제 주입. 그대로 두면 partial-attention 의 `#pragma omp parallel` 이 1-thread 로 직렬화. 세 C++ helper (`partial_attention_portable.cpp` / `_avx512.cpp` / `_amx.cpp`) 모두 `vllm_partial_attn_thread_count()` 함수를 두어 `sched_getaffinity` baseline + `VLLM_PARTIAL_ATTN_THREADS` env override (있으면 `min(env, baseline)` 으로 clamp 하여 oversize 설정으로 EAGAIN 재발 방지) 로 thread 수 결정, parallel pragma 의 `num_threads(...)` clause 에 사용. 캐시 함수-static 이라 호출당 1 회 평가 | **구현 완료** — 세 helper syntax 클린, prod simd_verify / cold_verify 통과 |
| (b''') Python dispatch hot-path 캐싱 | `forward_partial_with_lse` 가 매 호출마다 `select_isa_path()` 를 재평가, 그 안에서 `_has_amx()` / `_has_avx512()` 가 `cpuinfo.get_cpu_info()` 를 매번 새로 돌려 호출당 5+ 초의 wrapper 오버헤드를 만들었음 (prod profile 평균 5462 ms / call vs 내부 C++ 9.4 ms / call → 격차 5.45 초가 모두 Python dispatch 에서 새고 있었음). `_read_cpu_flags_once()` 함수-속성 캐시 + `select_isa_path._cache` 함수-속성 캐시 + `pin_threads_to_local_numa` 의 `_pin_fast_done` 모듈-level bool 로 hot path 호출당 비용을 ~ 수 µs 수준으로 단축 | **구현 완료** — prod 회차 로그상 wrapper time 5462 ms → 12 ms (median), gap median ≈ 0 |
| (c) 큰 cpu_bytes_to_use 의 분할 | 본 단계는 (a) 의 numa_set_preferred 가 worker 별로 다른 node 를 선호하게 만들면 자연스럽게 해결. 명시적 분할 없이 8 worker 의 first-touch 가 4+4 로 흩어지면 사실상 두 socket 에 균등 배치. 추가 분할이 필요할 만큼 worker 간 메모리 압박이 비대칭으로 떨어지는 케이스가 있으면 후속으로 검토 | **현 단계 불필요** — (a) 의 per-worker preferred node 만으로 충분 |
| (d) 검증 | [`TST_002`](TST_002.md) sweep 결과의 socket-local vs cross-socket 비교. `numactl --hardware` + `numastat -p <pid>` 로 페이지 분포 확인 | **부분 진행** — `eval/run_prod_cold_verify.sh` 가 `monitor.py` 로 CPU/GPU 시계열 캡처하며 cold path 발화 검증 통과 (prod cold_verify @ b3407a3bd9: 50/50 success, cold path 40 회 발화, CPU avg 30.9% / GPU avg 29.8%). 풀 throughput sweep + cross-socket 페이지 분포 확인은 별도 prod 회차로 진행. 회귀 fix 흐름 — partition / OMP num_threads / dispatch 캐싱 / device-mismatch — 은 [`PLN_001_TSK_002_02_overlap_fix_log.md`](PLN_001_TSK_002_02_overlap_fix_log.md) 와 통합 기록 |

### 1.3 · 비범위

- 정확도 (TST_003 — D-i / D-ii) — NUMA 무관, 본 TSK 와 별개
- multi-host 분산 KV — IDE_006 의 단일 호스트 가정 안에서만 적용
- GPU↔CPU 전송 자체의 NUMA-aware 화 (`cudaHostRegister` 의 NUMA 힌트) — 별도 후속 검토. 본 TSK 는 CPU-side 만 다룸
