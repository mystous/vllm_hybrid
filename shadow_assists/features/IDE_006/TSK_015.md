## 1. TL;DR

vLLM 의 *mirror* 정책 (cold blocks 가 GPU + CPU 양쪽 잔류) 을 NEO 의 *request 단위 exclusive ownership* 으로 변경. 한 request 의 KV 는 GPU 또는 CPU 한 쪽에 *exclusive* 로 존재. swap-in/out 은 *atomic* — source 에서 free + dest 에 alloc.

이 정책 변경이 NEO 의 *batch 확장* 을 가능하게 하는 fundamental 영역. mirror 라면 GPU 공간이 cold blocks 로 점유되어 batch size 못 키움.

### 3.2 · Phase 4.4 (vLLM 통합) sub-step 분할 — multi-day 작업 design

**핵심 어려움**: vLLM 의 기존 `_preempt_request` 와 NEO 의 swap_out 은 *의미가 다름*.

| 항목 | vLLM `_preempt_request` (`scheduler.py:961`) | NEO swap_out |
|---|---|---|
| KV 처리 | `kv_cache_manager.free` (폐기 — recompute 또는 swap_to_disk) | CPU 에 보관 (NEO 의 `cpu_decoding_q` source of truth) |
| `request.status` 전이 | `RUNNING → PREEMPTED` | `RUNNING → SWAPPED_OUT` (신규 enum 필요) |
| `num_computed_tokens` | **0 으로 reset** (재 prefill 가정) | **보존** (이미 생성한 token 까지) |
| 다음 위치 | `waiting.prepend_request` | NEO `cpu_decoding_q` (vLLM 의 waiting 으로 가지 *않음*) |
| `spec_token_ids` | 폐기 | 보존 |
| `num_preemptions` | +1 | unchanged (preempt 가 아님) |

따라서 `_preempt_request` 직접 호출 불가. 4.4.a / 4.4.b helper 는 NEO 식 의미를 가진 **별도 transition** 으로 작성.

**상태 머신 (제안)**:
```
RUNNING ─[NEO swap-out]──► SWAPPED_OUT ─[NEO swap-in]─► RUNNING
   ▲                            │
   │                            └──[finish_requests]──► FINISHED
   │
   └─ vLLM 기존 PREEMPTED 와 *별개 trace* (waiting 에 들어가지 않음)
```

**호환 검토 포인트**:
1. `Scheduler.schedule()` 의 running 순회 — `SWAPPED_OUT` 은 skip (NEO sibling 만 다룸).
2. `update_from_output` 의 model output 분배 — SWAPPED_OUT 한 req 는 본 step 의 model output 에 없음 (그 step 에 schedule 되지 않았으므로).
3. `finish_requests` — SWAPPED_OUT 상태에서도 abort 가능해야 함 (KV 는 CPU buffer 에서 free).
4. `KVCacheManager.allocate_slots` 의 swap-in 대응 — num_computed_tokens 보존된 채 다시 alloc 시 block reuse 가능 여부 확인.

**Phase 4.4 의 land 순서 (4 단계)**:
1. **4.4.a** — `_neo_swap_out` helper 만 단독 작성 + unit test. EngineCore 호출은 안 함. 회귀 0.
2. **4.4.b** — `_neo_swap_in` helper 작성 + unit test. EngineCore 호출 안 함. 회귀 0.
3. **4.4.c** — EngineCore 의 `step()` 에서 `update_from_output` 직전 hook 추가. `VLLM_NEO_KV_FREE=1` opt-in. 환경변수 OFF default 라서 prod 회귀 0.
4. **4.4.d** — `VLLM_NEO_FORCE_CDEC=1 VLLM_NEO_KV_FREE=1` 양방향 dev smoke. token equality 확인. 정상 동작 확인 후 4.5 / 본격 prod 활성 검토.

### 3.3 · Phase 4.5 (cdec dispatch) 의 architectural mismatch 발견 (2026-05-01)

Step3.2.b/c 의 *모든 인프라* (CommonAttentionMetadata.neo_cdec_token_slice/seq_slice/req_ids + ForwardContext fork + unified_attention_with_output dispatch hook + NeoCpuKvBuffer 연동) land 후 dev smoke (Qwen-1.5B + `VLLM_NEO_FORCE_CDEC=1 VLLM_NEO_KV_FREE=1 VLLM_NEO_FORCE_CDEC_DISPATCH=1`) 검증 시도 → **dispatch hook 의 try block 진입 zero** 발견.

**Root cause** — Phase 4.4.c 의 `_handle_neo_swaps` 가 cdec req 를 `running.remove` + status `SWAPPED_OUT` 으로 변경. 결과:

```
NEO scheduler:
  batches[0] = [prompt_req (gprf)]    ← vLLM running 에 있음
  batches[1] = [cdec_req]              ← vLLM running 에 *없음*

vLLM 측:
  input_batch.num_reqs = 1  (prompt_req 만)
  input_batch 에 cdec_req 의 token 자체가 없음

gpu_model_runner.execute_model fork branch:
  contiguous-from-0 검사: boundary(1) + len(neo_subs[1])(1) = 2 ≠ num_reqs(1)
  → fork 발화 안 함 → ForwardContext.neo_cdec_token_slice 영영 None
  → unified_attention_with_output dispatch hook 진입 zero
```

즉 **Phase 4.4 의 `running.remove` 가 Phase 4.5 의 dispatch path 를 dead code 로 만듦**. NEO 의 정통 architecture 는 *cdec req 도 forward 에 참여* (Q embedding 만 GPU 에서 생성, attention backend 단계에서만 CPU pacpu 로 분기) — 이게 본질적 정합 영역.

**해결 path (multi-day)**:
1. `_handle_neo_swaps` 의 `running.remove` 제거 — cdec req 를 *running 에 유지* 하되 status `SWAPPED_OUT` 표시
2. vLLM `Scheduler.schedule()` 갱신 — `SWAPPED_OUT` 상태 req 도 input_batch 에 포함 (Q embedding generation path 유지)
3. `KVCacheManager` 갱신 — `SWAPPED_OUT` req 의 GPU blocks 는 free 됐으나 attention 단계는 backend 가 NeoCpuKvBuffer 의 CPU view 사용
4. fork branch 의 contiguous check 갱신 — cdec req 도 num_reqs 에 포함하도록 정합

위 4 단계가 새 sub-step (`TSK_015.Phase4.5.B-*` 또는 새 영역) 으로 분할 적재 필요. **Phase 4.4 의 design 자체 갱신 영역** — 단순 hook 추가 외 vLLM core (Scheduler / KVCacheManager / input_batch) 갱신.

**현 상태** (Step3.2.b + Step3.2.c.1~c.8):
- 인프라 100% land — 회귀 zero (env OFF default)
- dispatch path 자체는 정확 — Phase 4.4 의 design 갱신 후 즉시 활성 가능
- timeout 의 또 다른 원인 — `FORCE_CDEC_DISPATCH=1` 의 swap_in skip + 매 step 새 victim → running 고갈 → engine hang. dev hook 자체의 정합 영역 (별도 fix)

## 4. 알고리즘 reference

[`NEO_code_deepdive.md`](NEO_code_deepdive.md) §5.

핵심 invariant — *exclusive*:

```python
# 어느 시점이든:
for req in all_requests:
    has_gpu = req in gpu_block_manager.allocated
    has_cpu = req in cpu_block_manager.allocated
    assert has_gpu ^ has_cpu  # XOR — 정확히 한 쪽
```

`_initiate_swap` 의 atomic:

```python
def _initiate_swap(reqs, is_swap_out):
    src = gpu_block_manager if is_swap_out else cpu_block_manager
    dst = cpu_block_manager if is_swap_out else gpu_block_manager
    src_pids = src.free(reqs)              # source 의 block 회수
    dst_vids, dst_pids = dst.alloc(reqs)   # dest 에 alloc
    return src_pids, dst_vids, dst_pids
```

→ swap 의 *physical transfer* (PCIe) 는 별도 worker thread 에서. 본 TSK 의 영역은 *block table 의 ownership 갱신 atomic*.
