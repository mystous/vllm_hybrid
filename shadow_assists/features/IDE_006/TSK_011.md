## 1. TL;DR

매 layer 의 attention 단계에서:
- GPU 는 hot only FA 를 *항상* launch (cold 영역 제외)
- CPU 는 cold partial attention 을 best-effort 로 시작
- merge 직전 deadline (또는 cold result 의 ready 검사) 시점에:
    - cold ready → `merge_attn_states(hot, cold)` 로 layer 출력
    - cold 미도달 → GPU 가 cold 포함 full FA 를 추가 호출, hot only 결과는 폐기, full FA 결과로 layer 출력

수학적 동등성은 README §4.1.1 의 LSE-rescaling 정의가 보장 (fallback 결과 = partition 합산 결과).

> **2026-04-29 의미 재정의**: 본 TSK 의 *blocking deadline* (`future.result(timeout=deadline_s)`) 은 TSK_009 fix v4 의 *non-blocking poll* (`future.done()`) 으로 대체됨 (사용자 framing — "CPU 가 GPU 의 작업을 절대 방해해서는 안됨"). 본 TSK 의 fallback path (`_fallback_full_fa_paged`) 는 fix v4 의 *paged FA full inplace* 로 통합되어 GPU FA 호출 = 항상 1번 보장. 즉 본 TSK 의 deadline 메커니즘 자체는 *non-blocking 으로 재구현*. fallback path 의 코드는 보존 (예외 안전망 only — TSK_012 적용 후 partition path 발화 안 함이라 발화 영역 자체 없음).

## 3. 변경 범위

### 3.1 · 흐름 변경

```
[현재 §4.5c 흐름]
  hot FA launch
  cold path issue (sync)
  wait_stream(cold_stream)
  merge_attn_states(hot, cold)

[목표 §3.1 3차 재정의 흐름 — TSK_011 본문 (2026-04-28)]
  hot FA launch (cold 영역 제외)
  cold path issue (async, deadline budget 설정)
  await cold result with *blocking* deadline:                # ← fix v4 에서 변경
    if ready: merge_attn_states(hot, cold)
    else:     GPU full FA on (hot + cold), hot result 폐기   # ← GPU FA 2회 호출

[fix v4 — TSK_009 (2026-04-29)]
  cold path issue (async, ThreadPoolExecutor)
  *non-blocking* future.done() poll
    if done:
        hot subset paged FA + merge_attn_states(hot, cold)   # GPU FA 1회 (hot subset)
    else:
        future.cancel() + paged FA full inplace + return     # GPU FA 1회 (full)
                                                              # ← hot path 호출 자체를 done 분기로 지연
```

→ fix v4 가 본 TSK 의 *blocking deadline* 을 *non-blocking poll* 로 대체. 둘 다 paged FA 1번 호출 보장 — 사용자 framing ("CPU 가 GPU 작업 방해 안 함") 정합.
