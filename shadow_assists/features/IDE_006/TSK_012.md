## 1. 발급 배경 — TSK_011 sweep 의 결정적 발견

`TSK_011` §4.5 prod sweep (2026-04-28) 결과:

| 시나리오 | fallback # | worst_lp | worst_ppl | d_ii |
|---|---|---|---|---|
| 비활성 (TSK_011 코드 없음) | 0 | 3.428 | 0.2423 | 8/30 |
| deadline=100ms (fallback 100%) | 40 | 3.429 | 0.2299 | 9/30 |
| deadline=1000ms (fallback 0%) | 0 | 3.428 | 0.2423 | 8/30 |

**deadline=1000ms 에서 fallback 0회** + **결과가 비활성과 bit-exact 일치** → partition path 자체가 baseline 과 발산. **deadline=100ms 에서 fallback 100%** + **partition 결과와 거의 같은 lp** → fallback path 도 같은 발산.

→ partition (`merge_attn_states(hot, cold_via_CPU_partial_attn_LSE)`) 와 fallback (`SDPA on hot+cold_via_CPU_paged_view_H2D`) 두 경로 모두 **cold KV 를 CPU page cache 에서 가져옴**. 반면 baseline 은 **OffloadingConnector 가 비활성** 상태로 모든 KV 가 GPU paged cache 상주 → 두 source 의 ULP-level 차이가 logprob 에 누적되어 lp ~3.43 이 됨.

본 TSK 는 그 source 차이를 닫는 작업.

## 3. 변경 범위

### 3.1 · 흐름 변경

```
[현재 (§4.5c reload-fallback) — prefill 만]
  prefill admission
    → OffloadingConnector 가 cold blocks 를 GPU paged cache 로 reload
    → standard hot FA 가 모든 KV 처리

[목표 — decode 영역도 동일]
  decode step 진입 전
    → 그 step 의 cold-bearing seqs 의 cold blocks 를 GPU paged cache 로 reload
    → standard hot FA 가 모든 KV 처리
    → IDE_006 cold path (partition / fallback) 자체가 *발화하지 않음*
       또는 partition path 가 *발화하더라도* 같은 source 사용
```

본질: cold path 가 발화 안 하면 baseline 과 동일 동작 → D-ii 자동 봉합. 단 throughput 측면 손실 (cold blocks 가 항상 GPU 에 있으므로 KV pool 압박 ↑, OOM 위험 ↑).

### 3.2 · 변경 파일 (예상 — 조사 후 확정)

| 파일 | 변경 |
|---|---|
| `vllm/distributed/kv_transfer/kv_connector/v1/offloading/scheduler.py` | `update_state_after_alloc()` 또는 별도 hook 에 decode admission 시점 cold→GPU reload trigger 추가 |
| `vllm/v1/core/sched/scheduler.py` | decode step entry 시점 cold blocks identification + reload spec |
| `vllm/v1/worker/gpu_model_runner.py` | reload completion sync 가 decode 영역까지 적용 (현재 §4.5c 는 prefill 만) |

`TSK_009` 와 함께 land 시 같은 hook 의 *admission prefetch (TSK_009)* + *decode reload (본 TSK)* 를 동일 patch 로 처리.

### 3.3 · 정확성 vs 처리량 트레이드오프

본 TSK 는 정확성 우선 — D-ii 봉합. cold blocks 를 GPU 에 reload 하므로:

- ✅ baseline 과 동일 source → D-ii 자동 봉합
- ❌ KV pool 압박 ↑ — cold blocks 가 GPU 에 다시 상주
- ❌ OffloadingConnector 의 cold-tier 메리트 (CPU DRAM 활용) 일부 무효화
- 보완: `TSK_005` (cross-layer pipeline) 와 결합 시 reload 비용을 forward 와 overlap 가능

이 트레이드오프 자체가 IDE_006 의 정의를 *재검토* 하는 영역 — long-context workload 에서 KV 가 GPU 에 안 들어가면 reload 가 불가능 (역설적). 즉 본 TSK 는 *짧은~중간 context* 영역에서만 의미. 진정한 long-context 에서는 다른 메커니즘 필요 (NEO-style sub-layer pipeline, IDE_004 영역).
