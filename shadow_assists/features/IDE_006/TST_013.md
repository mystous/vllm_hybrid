## 1. 의도

본 TST 는 두 차원을 검증한다:

- **§2 분석 plan 의 완결성** — TSK_013 의 산출물 (`PLN_001_TSK_013_neo_arch_survey.md` + 신규 TSK 발급 plan) 이 NEO 의 모든 핵심 메커니즘을 누락 없이 cover 하는지. 누락이 있으면 NEO 식 architecture 의 vLLM 적재 시 *동작 불완전* 위험.

## 2. 분석 plan 완결성 검증 (binding)

### 2.1 · NEO 의 핵심 모듈이 모두 매핑되었는가

TSK_013 §3 의 매핑표 + survey 본문이 다음 항목을 모두 다루는지 점검:

| 항목 | 검증 |
|---|---|
| Request scheduler (3 큐 + load-aware mode selection) | scheduler.py 의 모든 핵심 메서드가 vLLM 의 `Scheduler.schedule()` hook 에 매핑 |
| Sub-batch 동시 실행 | executor.py 의 SingleProcExecutor / RayExecutor 가 vLLM 의 model runner 에 매핑 |
| KV cache exclusive ownership | block_manager.py 의 admission / eviction / swap 정책이 vLLM 의 KVCacheManager 에 매핑 |
| GPU/CPU block swap | block_swapper.py 의 메커니즘이 vLLM 의 `cpu_gpu.py` 에 매핑 |
| Perf prediction (load-aware scheduling 입력) | perfpredictor.py + profiler.py 가 vLLM 의 신규 모듈로 매핑 |
| CPU attention kernel | pacpu/ ISPC + C++ kernel 이 IDE_006 의 AVX-512/AMX kernel 과 비교 + port 결정 |
| Sub-batch attention 의 모델 forward 통합 | worker/model.py 의 forward 가 vLLM 의 모델별 forward (Llama / Qwen / Mistral) 에 매핑 |
| Request / SubBatch 추상화 | structs.py 가 vLLM 의 request 추상화에 매핑 |

**Pass 기준**: 위 8 개 항목 모두에 대해 *NEO 의 코드 위치 + vLLM 의 hook 위치 + 변경 범위* 가 survey 본문에 명시되어야 함. 하나라도 누락되면 본 TST 미통과 → TSK_013 추가 분석.

### 2.2 · 신규 TSK 5 개의 변경 범위가 명확한가

§4 의 5 개 신규 TSK (Request scheduler / KV exclusive / Asymmetric pipelining / Load-aware scheduling / CPU kernel port) 각각에 대해:

| 검증 차원 | 기준 |
|---|---|
| 변경 파일 list | vLLM 코드의 정확한 파일 경로 (line 번호까지는 아니어도 함수 명세) |
| 의존 관계 | 다른 신규 TSK 와의 선후 관계 명시 |
| 검증 영역 | 각 신규 TSK 의 검증 게이트 (TST) 가 어떤 기준을 측정할지 |
| 회귀 영역 | 변경이 영향을 미치는 기존 vLLM path (예: vanilla GPU-only path 가 깨지지 않는지) |

**Pass 기준**: 5 개 신규 TSK 모두에 대해 4 차원 명시. 하나라도 누락되면 본 TST 미통과.

### 2.3 · IDE_006 의 기존 8 개 적용 TSK 와의 정합

[`NEO_redesign.md`](NEO_redesign.md) §2 의 표 (✅ 4 / 🔶 2 / ❌ 2) 가 신규 TSK 5 개의 변경 범위와 정합인지:

| 기존 TSK | NEO 식 적용 시 | 신규 TSK 의 변경 범위와 정합? |
|---|---|---|
| TSK_001 (kernel 코어) | 🔶 부분 사용 | 신규 TSK_018 (CPU kernel port) 가 TSK_001 의 코어 재사용 명시 |
| TSK_002 (hot/cold partition) | 🔶 부분 사용 | 신규 TSK 들 중 어느 것도 hot/cold partition 사용 안 함 명시. Q D2H stream 분리 인프라만 TSK_016 (Asymmetric pipelining) 에서 재활용 |
| TSK_003 (SIMD) | ✅ | 신규 TSK_018 에 그대로 통합 |
| TSK_004 (NUMA) | ✅ | 신규 TSK_018 에 그대로 통합 |
| TSK_007 (GQA 옵션 A) | ✅ | 신규 TSK_018 에 그대로 통합 |
| TSK_009 (non-blocking dispatch) | ❌ | 신규 TSK 들에서 사용 안 함 명시 |
| TSK_010 (sub-batching) | ✅ | 신규 TSK_018 또는 TSK_016 에 그대로 통합 |
| TSK_011 (fallback) | ❌ | 신규 TSK 들에서 사용 안 함 명시 |

**Pass 기준**: 모든 항목에 대해 신규 TSK 의 변경 범위가 정합. 하나라도 충돌하면 본 TST 미통과 → TSK_013 plan 재작성.

## 3. (선택) NEO 자체 reproduce — throughput gain 재현 검증

### 3.1 · 환경

NEO repo 의 reproduce 스크립트는 *AWS T4 (g4dn.4xlarge)* 또는 *A10G (g5.16xlarge)* 환경 가정. IDE_006 의 prod target (Xeon SPR + H100×8) 에서 reproduce 시 다음 영역 변경 필요:

- `evaluation/configs/config-t4-7b.json` 또는 `config-a10-8b.json` 의 model_path / hardware spec 갱신
- `pacpu/build.sh` 의 model-name + tensor-parallel-degree 갱신
- Intel ISPC 1.23 설치 (`sudo snap install ispc --channel latest/edge`)
- g++ 두 버전 (≥ 13 + < 13) 설치

또는 *별도 평가 환경* (T4 또는 A10G) 에서 NEO 의 본래 결과를 *그대로* reproduce.

### 3.2 · Pass 기준

| reproduce 환경 | NEO 의 reported gain | 본 TST 의 목표 |
|---|---|---|
| T4 (16 GB) + LLaMa-2-7B | up to 7.5× (output sweep 750%) | reproduce gain ≥ 5× (NEO 의 60% 수준) |
| A10G (24 GB) + LLaMa-3-8B | up to 26% (강한 CPU 시 79.3%) | reproduce gain ≥ 15% |
| H100 (80 GB) + LLaMa-70B + TP=8 | up to 14% (paper claim, reproduce script 없음) | NEO repo 위 직접 측정 — gain ≥ 8% 또는 reproduce 환경 영역 외 (별도 PoC TSK 필요) |

**Pass 기준**: 위 환경 중 *최소 1 개* 에서 NEO 의 60% 수준 gain 재현. 재현 불가능이면 NEO 식 architecture 의 vLLM 적재의 *진입 의의* 자체가 흔들림 → 사용자 결정 영역.

### 3.3 · 비-필수성

§3 은 *선택* — NEO 의 reproduce 가 안 되더라도 §2 의 분석 plan 완결성만 통과하면 본 TST 는 *부분 통과* 로 인정. NEO 의 결과를 paper / repo README 에 의존하여 진행 결정 가능.
