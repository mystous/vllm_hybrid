## 1. TL;DR

NEO 의 `_forward_pipeline` — 두 SubBatch 가 *layer 단위 ping-pong* 으로 동시 실행. GPU 가 batches[0] 의 attention + linear 를 진행하는 동안 CPU 가 batches[1] 의 attention 을 *별도 thread* 로 진행. layer 마다 두 batch 의 작업이 교차.

이게 IDE_006 의 4 차 재정의의 *시그니처 메커니즘* — **Q dependency dilemma 회피의 fundamental 영역** (서로 다른 request 가 같은 layer 의 다른 단계를 동시 처리 → GPU 가 그 request 의 attention 을 *안 하기로 결정* → CPU 결과가 유일한 결과).

## 4. 알고리즘 reference

[`NEO_code_deepdive.md`](NEO_code_deepdive.md) §4.

### 4.1 · *Layer offset* fundamental 메커니즘 (핵심)

NEO 의 `forward_double` 은 한 호출에서 *두 batch 가 서로 다른 layer 의 attention* 처리:
- batch[1] : layer **i** attention (cur_stage=0)
- batch[0] : layer **i+1** attention (cur_stage=1)

`cur_layer_id = (layer_id + cur_stage) % num_layers` 로 결정. 두 batch 가 *layer 1 만큼 어긋나 진행* — pipeline depth 1 with layer offset. 이게 attention output 의 layer 안 즉시 dependency 를 *cross-batch 로 분리* 하는 메커니즘.

vLLM 위 적재 시 `gpu_model_runner.execute_model` 의 신규 `_execute_pipeline` hook 에서 *layer offset 을 명시적으로 관리* 필요. Llama forward 의 stage 분할:
- `forward_first_stage(embeddings, batches)` → `(q1, k1, v1)` of batch[1] layer 1
- `forward_double × (N-1)` — 매 호출마다 batch[1] 의 layer i + batch[0] 의 layer i+1
- `forward_last_stage(q1, k1, v1, batches)` → batch[1] 의 layer N attention + 두 batch post

### 4.2 · `_attention` 의 3-way dispatch ([`NEO_code_deepdive.md`](NEO_code_deepdive.md) §4.5)

한 SubBatch 안에 prefill / GPU decode / CPU decode 가 섞일 수 있으므로 attention dispatch 가 3-way:

```python
if batch.num_prefs > 0:    # 1. prefill — flash_attn (Ampere+) 또는 prefill_attention
    flash_attn_cuda.varlen_fwd(...)
if batch.num_gdecs > 0:    # 2. GPU decode — paged attention
    paged_attention(...)
if batch.num_cdecs > 0:    # 3. CPU decode — TSK_018 의 CPU kernel (IDE_006 의 AVX-512/AMX)
    qkvtr_e.synchronize()  # Q D2H 완료 대기
    cpu_attention_kernel(q_cpu, k_swap, v_swap, cpu_block_table, o_cpu)
    o[-num_cdecs:].copy_(oc, non_blocking=True)  # output 만 GPU 로
```

vLLM 의 `flash_attn.py` 가 본 분기를 sub-batch 인지하도록 변경. CPU kernel 호출 사이트가 TSK_018 의 진입점.

### 4.3 · `_transfer_qkv` Q/K/V D2H ([`NEO_code_deepdive.md`](NEO_code_deepdive.md) §4.6)

`num_cdecs > 0` 이면 *별도 stream* 에서 Q/K/V 를 CPU 로 D2H. 끝나면 `qkvtr_e.record()`. `_attention` 가 진입 시 `qkvtr_e.synchronize()`. 이 패턴은 IDE_006 의 TSK_002 §4.6 stream 분리와 정합 — *재활용 가능 인프라*.

### 4.4.0 · Step 5.4 wiring 통과 + forward-context fork blocker (2026-04-30)

dev 머신 (RTX 3090 + Qwen2.5-1.5B-Instruct) 에서 `VLLM_NEO_FORCE_PIPELINED=1` 로 force-split 실행 결과:

| 진단 chain | 발화 신호 | 의미 |
|---|---|---|
| scheduler→runner attachment | `[NEO-DEBUG] execute_model gate: 2 sub-batch(es), req-id counts=[1, 1]` | Step 5.1 + 5.2 + 5.3 통과 |
| dual forward branch entry | `[NEO-DEBUG] _model_forward: attempting dual forward (sub_batches=[1, 1])` | Step 5.4 분기 ✓ |
| dual forward 결과 합산 | `[NEO-DEBUG] _model_forward: dual forward returned 2 tensors, max\|out0-out1\|=4.000e+01` | Step 5.4 list→tensor 합산 도달 ✓ + divergence 검출 |
| 무회귀 fallback | `[NEO-DEBUG] _model_forward: dual forward fell back, exception was RuntimeError: dual forward diverged ... KV cache cross-contamination detected; forward-context fork required (Step 5.5)` | divergence 임계 (1e-3) 초과 시 vanilla 로 깨끗하게 fallback |
| 최종 회귀 검증 | `token-id equality (vanilla vs NEO): PASS` | smoke 통과 |

**diagnosis**: 본 stage 의 `forward_neo_pipelined` 호출은 양쪽 sub-batch 에 *동일* `positions` / `embeddings` / `slot_mapping` 을 공유. 동일 입력이면 두 forward 의 출력이 **bit-exact 동등** 해야 하는데 `max|out0-out1| = 40.0` 으로 발산. 원인 = vLLM `Attention` 모듈의 KV cache write:

1. 1차 호출 (batch[0]) — `slot_mapping` 의 slot 에 K, V 쓰기
2. 2차 호출 (batch[1]) — 동일 slot 에 K, V 다시 쓰기 + 이미 1차가 쓴 prefix slot 을 attention 으로 read → numeric 발산

→ **Step 5.5 = forward-context fork**: sub-batch 별로 *disjoint* `slot_mapping` / `cu_seqlens` / `block_tables` 분할 후 각자 자기 KV pool 영역에만 쓰도록 변경. 이게 NEO §5 (KV exclusive ownership, TSK_015) 의 vLLM 적재 영역.

### 4.4 · 메인 loop

핵심 — layer 단위 ping-pong:

```python
# vllm/v1/worker/sub_batch_executor.py
def forward_pipeline(batches, embeddings):
    # Stage 0 — 마지막 layer 의 first stage 만 (= QKV projection of batches)
    q1, k1, v1 = transformer_layers[-1].forward_first_stage(embeddings, batches)

    # Layer ping-pong — N-1 회 반복
    for layer in transformer_layers[:-1]:
        q1, k1, v1 = layer.forward_double(q1, k1, v1, batches)
        # forward_double 안:
        #   - GPU: batches[0] 의 attention + post + 다음 batch 의 input projection
        #   - CPU: batches[1] 의 attention (별도 thread 또는 ThreadPoolExecutor)
        #   - sync: layer 끝에 두 batch 의 결과 합침

    # Last stage
    embeddings = transformer_layers[-1].forward_last_stage(q1, k1, v1, batches)
    return embeddings
```

GPU/CPU 동시 실행 hook (`forward_double` 내부 의사 코드):

```python
def forward_double(q1, k1, v1, batches):
    # GPU 작업 (current stream)
    fut_gpu = gpu_stream.submit(
        lambda: attention_post(batches[0], ...) + qkv_proj_next(batches[1])
    )
    # CPU 작업 (별도 thread)
    fut_cpu = cpu_executor.submit(
        cpu_attention, batches[1].cdec_reqs, q_for_batch1, k_cache_cpu, v_cache_cpu
    )
    # 두 결과 sync
    out_gpu = fut_gpu.result()  # GPU stream wait
    out_cpu = fut_cpu.result()  # CPU thread join
    return q_next, k_next, v_next  # 두 batch 의 다음 layer 입력
```
