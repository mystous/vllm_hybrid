**결과**:
- engine dead 없음 — thread-local OMP fix (`omp_get_max_threads()` 우선 + Python `omp_set_num_threads` ctypes) 정상

- D-ii 동일 (worst_lp 3.65 / d_ii 91/100) — sub-batching 이 정확도 영향 없음

**미진행 영역** (1.1 per-core util / 1.3 cross-socket 트래픽) — wall-time 측정이 negative 라 추가 측정 가치 작음. 코드 인프라는 보존 (default off, 환경 변수 활성 시만 동작).

## 1. 검증 차원

### 1.1 · per-core utilization

prod 머신에서 cold path firing 동안:

```bash
mpstat -P ALL 1
# 또는
pidstat -t -p <vllm_worker_pid> 1
```

| 모드 | per-core %CPU |
|---|---|
| baseline (TSK_004 NUMA bind, 단일 OMP team) | TSK_004 partition 의 14 cores (TP=8 가정) 만 80%+ 사용, 나머지 0% |
| TSK_010 활성 (multi-team / sub-batching) | 같은 14 cores + sub-batch 가 활용하는 추가 cores (NUMA 안에서) 도 활용 ↑ |

contract: TSK_010 활성 시 *active core 수* 가 baseline 대비 ↑.

### 1.2 · sub-batch latency

batch 를 N sub-batch 로 split 후 각 sub-batch 의 wall time 측정. 단일 batch (baseline) wall time 과 비교:

```
single batch wall time:  T_single
N sub-batches wall time: T_max(sub_i)  # 동시 진행이라 max
```

contract: T_max(sub_i) < T_single (multi-team 이 진짜 동시 진행하면 max < single).

### 1.3 · cross-socket 트래픽 0 검증

`numactl --hardware` + Linux perf event `mem_load_uops_l3_miss_retired.remote_dram` 로 *cross-socket DRAM access* 빈도 측정. TSK_004 NUMA bind 가 작동하면 0 또는 매우 작음.

multi-team 활성 시에도 sub-batch 가 NUMA core slice 안에서만 실행되어 cross-socket 0 유지 — 이게 깨지면 multi-team 이 NUMA bind 를 깬 것.

### 1.4 · numerical equivalence

multi-team 결과가 단일 team 결과와 atol/rtol 통과.

## 2. Pass / Fail 기준

| 단계 | 기준 |
|---|---|
| 1.1 utilization | active core 수 (baseline) < active core 수 (TSK_010) |
| 1.2 sub-batch latency | T_max(sub_i) / T_single < 0.7 (30% 이상 단축, 진짜 동시 진행 입증) |
| 1.3 cross-socket | remote_dram access 빈도 ≤ baseline 의 1.1 배 (NUMA bind 보존) |
