## 1. 검증 차원

### 1.1 · 단위 — Scheduler 의 6 단계 algorithm invariant (binding)

| 검증 항목 | 기준 |
|---|---|
| swap-in 과 swap-out 동시 발생 | `assert not swp_out or not swp_in` |
| FCFS prefill 분류 | 한번 CPU 로 간 prefill 후속도 CPU |
| 두 SubBatch 의 합집합 = 전체 request 후보 | union check |
| ScheduleBudget 침범 | `budget.overspent == False` (정상 종료 시) |

### 1.2 · 단위 — `mode_selector._decide_mode_and_gen_batch` 정확성

| 검증 항목 | 기준 |
|---|---|
| `_get_remains[j]` 계산 | `remains[j] == batches[j^1].linr_T + batches[j].pref_T + batches[j].gdec_T - batches[j].cpu_time` |
| pipelined vs sequential rate 비교 | 두 mode 의 rate 정확 계산, 큰 쪽 채택 |
| `min(remains) < 0` 시 추가 cdec skip | invariant 유지 |

### 1.3 · 회귀 — vanilla GPU-only 무회귀

CPU running 큐가 비어있을 때 (= NEO 식 disable) vLLM 기존 scheduler 와 *bit-exact* 동등 동작:

| 시나리오 | 기준 |
|---|---|
| Llama-3.3-70B + TP=8 + max-prompts=30 + cold-tier OFF | wall-time 기존 ± 5% |
| 단위 테스트 `tests/v1/core/sched/test_scheduler.py` | 모두 통과 |

### 1.4 · heuristic 측정 — pipelined vs sequential 결정 분포

prod 환경에서 mode selection 의 *실측 분포*:

| 시나리오 | 측정 |
|---|---|
| input_heavy (15360/1024) | pipelined % |
| output_heavy (1024/15360) | pipelined % |
| equal (8192/8192) | pipelined % |

→ NEO 의 정량 결과 (T4 7.5× / A10G 26%) 와의 *환경 비교* 영역.
