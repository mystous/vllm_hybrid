## 1. 의도

`TSK_011` sweep 으로 입증된 *fallback 만으로 D-ii 봉합 불가능* 의 후속 검증. `TSK_012` 적용 후:

- cold blocks 가 *진짜 GPU evict* 되고 decode 시점에 reload 가 발화. attention 은 standard hot FA 가 모든 KV 처리. partition path 발화 안 함.
- *D-ii 봉합* 단일 목표 검증.
- (선택) capacity 차원 — KV pool overflow 영역에서 vanilla 대비 동시 in-flight request 수 향상 가능성.

TSK_005 기각 후 (`Q dependency dilemma`) 본 TST 가 IDE_006 의 *진짜 가치 영역 검증 게이트* 의 위치를 가짐. invariant 2 (CPU 활용 향상) 는 본 TSK 의 목표 영역 *밖* — IDE_006 partition path 자체가 발화 안 함.

### 2.2 · (선택) capacity 회차 — KV pool overflow 영역

KV pool overflow 발현 영역에서 vanilla mirror 대비 본 TSK 의 진짜 evict 가 capacity (수용 request 수) 확장에 기여하는지 측정. TSK_009 validation wrapper 재사용:

```
모델:        meta-llama/Llama-3.3-70B-Instruct
TP:          8
max-prompts: 100 (1.6M tokens, ~24% spill 발현 영역)
max-tokens:  16384 - input_len
logprobs:    0 (wall-time 측정만)
시나리오:    input_heavy / output_heavy / equal
모드:        B (cold-tier ON, IDE_006 OFF) vs C (TSK_012 적용, IDE_006 partition path 발화 안 함)
wrapper:     eval/run_prod_tsk009_validation.sh (재사용)
```

이 회차는 *vanilla 의 KV pool overflow 시 동작* (preemption / reject) 과의 비교가 핵심. 단 vLLM 의 mirror 정책의 정확한 동작은 *Phase 0 hook 영역 조사* (TSK_012 §5.1) 결과로 확정.

### 3.3 · reload event timeline

decode step 에 cold blocks 의 reload trigger 시점 (host timestamp) + completion 시점 (CUDA event query) 분포 기록. *forward 진입 시점에 reload 가 완료되어 있는 비율* 이 ≥ 90% 여야 함 — 그렇지 않으면 race window 가 열려 정확도 회귀.

### 3.4 · throughput tradeoff (per-request wall-time)

```
baseline (cold-tier OFF):                       ~164s (TSK_009 fix v4 검증 기준)
TSK_011 split_on (deadline=100ms / fb=100%):    187s (+14%)
TSK_011 split_on (deadline=1000ms / fb=0%):     216s (+32%)
TSK_009 fix v4 (cold-tier ON, IDE_006 OFF B):   200s (+22% — vLLM 자체 cold-tier)
TSK_009 fix v4 (cold-tier ON, IDE_006 ON C):    220s (+34%)
TSK_012 split_on (decode reload, partition off): ?  목표 = B 와 동등 (+22%) ± reload PCIe 비용
```

본 TSK 는 partition path 발화 안 함 → wall-time = B (cold-tier ON, IDE_006 OFF) + reload PCIe 비용. 단 cold blocks 가 진짜 evict 되면 KV pool 압박 완화 → 다른 영역에서 회복 가능. **per-request 차원에서는 vanilla (B) 보다 항상 느림이 fundamental** — reload PCIe 추가 비용.

### 3.5 · (선택) capacity 차원

KV pool overflow 영역 (max-prompts=100) 에서:

- vanilla (B) 의 *수용 request 수* 와 *preemption 발생 횟수* 측정
- TSK_012 적용 후 *수용 request 수* 와 *preemption 발생 횟수* 측정
- 진짜 evict 가 cold blocks 의 GPU 공간을 free 하므로 동시 in-flight request 수가 증가하는지 확인

목표 = TSK_012 의 capacity 가 vanilla 보다 ≥ 1.0× (= 동등 이상). 단 vLLM 의 mirror 정책의 KV pool overflow 동작이 *preemption 발생* 인 경우에만 본 메트릭이 의미 (mirror 가 GPU 공간 점유 안 하면 capacity 가치 = 0).
