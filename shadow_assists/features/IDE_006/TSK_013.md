## 1. TL;DR

NEO 는 자체 LLM 엔진 (`swiftllm`) — vLLM 의 fork 가 아니므로 *그대로 적용 불가*, *port / 모방* 영역. 본 TSK 는 NEO 의 핵심 모듈을 식별하고 vLLM 의 어느 hook 에 매핑되는지 정합표 + 적용 plan 을 산출한다. 코드 적재 자체는 본 TSK 의 영역 *밖* — 산출된 plan 에 따라 신규 TSK 들 (Request scheduler / KV exclusive / Asymmetric pipelining / Load-aware scheduling / CPU attention kernel port) 로 분할 적재.

> **차용 정책 (2026-04-29 사용자 결정)**: **알고리즘만 차용** — NEO 의 코드를 *직접 복사하지 않고* vLLM 위에 재구현. NEO 가 Apache 2.0 라이선스이므로 호환 가능하나 attribution 은 본 TSK 본문 + 신규 TSK 본문 + 코드 주석에 명시. 깊은 코드 dive 결과는 별도 문서 [`NEO_code_deepdive.md`](NEO_code_deepdive.md) 에 *언어 중립 의사 코드* 로 적재 — 향후 논문 영역의 reference 자료로도 사용.

### 2.2 · NEO 의 hardware / dependency 요건

- Python ≥ 3.10, PyTorch ≥ 2.4
- g++ 두 버전 필요 (≥ 13 for CPU kernel + < 13 for NVCC version check)
- **Intel ISPC compiler == 1.23** (CPU attention kernel)
- **AVX2 support** — IDE_006 의 prod target (Xeon SPR AVX-512 + AMX) 보다 보수적
- 평가 장비: AWS T4 (16 GB) + 8 cores Xeon P-8259CL / A10G (24 GB) + 2n cores EPYC 7R32

## 3. NEO 의 핵심 모듈 vs vLLM hook 매핑

| NEO 모듈 | 역할 | vLLM 매핑 hook | 매핑 난이도 |
|---|---|---|---|
| `swiftllm/server/scheduler.py` (17 KB) | request scheduler — prefill waitqueue / GPU decode runqueue / CPU decode runqueue 의 3 큐 + load-aware mode selection (batch picking) | `vllm/v1/core/sched/scheduler.py` 의 `Scheduler.schedule()` + `update_state_after_alloc()` | 중 — vLLM scheduler 에 *3 큐* 개념 신규 도입 |
| `swiftllm/server/executor.py` (3.4 KB) | SingleProc / Ray executor + `do_one_iteration()` (sub-batch 동시 실행) | `vllm/executor/` (이미 Ray 있음) + `gpu_model_runner.py` 의 `execute_model()` | 중 — sub-batch 동시 실행 hook 신규 |
| `swiftllm/server/engine.py` (8.8 KB) | main engine loop | `vllm/v1/engine/llm_engine.py` | 낮음 — 엔진 진입점 변경 |
| `swiftllm/server/block_manager.py` (11.5 KB) | KV cache block manager — GPU-cache + CPU-cache 분리 + request 단위 ownership | `vllm/v1/core/kv_cache_manager.py` + `vllm/v1/kv_offload/` + `vllm/distributed/kv_transfer/kv_connector/v1/offloading_connector.py` | **높음** — vLLM 의 mirror 정책을 *exclusive ownership* 으로 fundamental 변경 |
| `swiftllm/server/profiler.py` (16 KB) + `swiftllm/perfpredictor.py` (6.7 KB) | perf prediction — load-aware scheduling 의 결정 heuristic 입력 | 신규 모듈 — `vllm/v1/core/sched/perfpredictor.py` (가칭) | 중 — NEO 의 heuristic 그대로 port |
| `swiftllm/worker/model.py` (14.9 KB) | model worker — forward pass + sub-batch | `vllm/v1/worker/gpu_model_runner.py` + 모델별 forward (`vllm/model_executor/models/llama.py` 등) | **높음** — sub-batch 두 개 동시 실행 hook 이 attention layer 안에 들어감 |
| `swiftllm/worker/block_swapper.py` (4.4 KB) | GPU/CPU block swap | `vllm/v1/kv_offload/worker/cpu_gpu.py` (이미 swap 있음) | 낮음 — 기존 메커니즘 위에 *exclusive 의 swap-in/out* 정책만 추가 |
| `swiftllm/worker/kernels/` | GPU kernel (자체 구현) | `vllm/v1/attention/backends/flash_attn.py` 등 (이미 vLLM 표준) | — vLLM 의 paged FA 가 NEO kernel 보다 성숙. 가져올 영역 적음 |
| `pacpu/` (CPU attention kernel ISPC + C++) | CPU 에서 attention 직접 처리 | `vllm/v1/attention/ops/cpu_partial_attention.py` + `csrc/cpu/partial_attention_*.cpp` (이미 IDE_006 TSK_001/003 으로 적재) | 중 — NEO 의 ISPC 코드와 IDE_006 의 AVX-512/AMX 코드 중 어느 쪽이 우위인지 비교 필요 |
| `swiftllm/structs.py` | Request / SubBatch / etc. 추상화 | `vllm/v1/request.py` + 신규 `SubBatch` 추상화 | 낮음 — 추상화 추가 |

## 4. 신규 TSK 발급 plan (5 개 영역)

본 TSK_013 의 분석 결과를 바탕으로 NEO 식 architecture 의 vLLM 적재를 다음 5 개 신규 TSK 로 분할 (실제 발급은 본 TSK 의 분석 완료 + 사용자 승인 후):

| 영역 | 가칭 | 변경 범위 | 의존 |
|---|---|---|---|
| (1) Request-level scheduler | `TSK_014` (가칭) | `vllm/v1/core/sched/scheduler.py` 의 3 큐 + load-aware mode selection | (5) perf predictor 선행 |
| (2) KV cache exclusive ownership | `TSK_015` (가칭) | `vllm/v1/core/kv_cache_manager.py` + `offloading_connector.py` 의 mirror → exclusive | 독립 |
| (3) Asymmetric pipelining (sub-batch 동시 실행) | `TSK_016` (가칭) | `vllm/v1/worker/gpu_model_runner.py` + 모델별 forward (Llama / Qwen / Mistral) | (1)(2) 선행 |
| (4) Load-aware scheduling heuristic | `TSK_017` (가칭) | 신규 `vllm/v1/core/sched/perfpredictor.py` + scheduler 통합 | (1) 선행 |
| (5) CPU attention kernel — NEO 의 ISPC vs IDE_006 의 AVX-512/AMX 비교 + port 결정 | `TSK_018` (가칭) | `vllm/v1/attention/ops/cpu_partial_attention.py` + `csrc/cpu/` | (3) 와 함께 |

(*가칭 — 실제 ID 발급은 id_registry "다음 부여 번호" 가져온 뒤 +1, 본 TSK 분석 완료 후*)
