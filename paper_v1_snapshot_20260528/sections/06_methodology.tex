\section{실험 방법}
\label{sec:methodology}

\algname 의 두 설계 결정---교차 도메인 화음과 박자 정렬---은 \emph{설계 원리}로 형식화되었다 (\S\ref{sec:metronome}).
본 절은 이 원리들의 경험적 정당화를 위한 \emph{분류군 기반 대표 검증} 방법을 기술한다.

\subsection{설계 공간의 5 분류군 Taxonomy}
\label{subsec:taxonomy}

CPU 자원 활용 기법은 자원의 사용 \emph{도메인}에 따라 다음 다섯 분류군으로 체계화된다.
각 분류군은 서로 다른 hardware/software 차원의 자원을 활용하며, 분류군 \emph{내부}의 기법들은 동일 도메인 자원을 경쟁한다.

\textbf{분류군 A --- 직접 커널 대체.}
GPU 의 일부 critical path 연산 (tokenizer, sampling, matmul, prefill) 을 CPU AVX-512 또는 AMX 커널로 직접 대체.
도메인: critical-path kernel (GPU 와 동일 시간 슬롯).

\textbf{분류군 B --- 시스템 수준 격리.}
OS/kernel 수준 자원 격리 (NUMA 고정, cgroup, transparent hugepages, taskset).
도메인: OS context-switch / cache / TLB.

\textbf{분류군 C --- Sub-step CPU 스케줄링.}
GPU step 내부의 sub-phase 경계 (attention vs.\ linear) 를 인지하여 그 사이의 idle 구간에 CPU 작업 burst.
도메인: step-internal scheduling.

\textbf{분류군 D --- 보조 워크로드 오프로딩.}
GPU 에 없는 추가 작업 (cold-KV 압축 해제, Jacobi LM-head, AMX draft head) 을 CPU 에 부담.
도메인: NEW workload (microbench-only feasibility).

\textbf{분류군 E --- 동시 보조 작업.}
GPU step 과 무관한 CPU side-channel 작업 (정규/분기 패턴 $\times$ 주기 변수) 을 별도 process 로 fire.
도메인: process-isolated CPU work.

\subsection{분류군별 대표 가설과 검증 protocol}
\label{subsec:hypotheses}

\algname 의 설계 결정은 다음 세 가지 가설로 분해된다.
각 가설은 한 분류군의 \emph{대표 기법}으로 검증된다.

\textbf{가설 H1 (교차 도메인 중첩):}
서로 다른 도메인의 두 기법을 stacking 하면 합산 효과가 단순 합에 가깝다.
같은 도메인 stacking 은 destructive interference 가 누적된다.
검증: 분류군 B 단독 (NUMA pin), 분류군 E 단독 (보조 작업), B$\oplus$E (cross-domain pair), B$^{4}$ (same-domain stack).

\textbf{가설 H2 (박자 정렬):}
CPU 보조 작업 주기 $\tcpu = k \cdot \tgpu$, $k \in \{2,3\}$ 의 정수배 정렬이 amortize 효율을 최대화한다.
검증: 분류군 E 의 $\tcpu \in \{10, 20, 100\}$\,ms $\times$ pattern $\in \{$regular, branchy$\}$ 변수.

\textbf{가설 H3 (분류군별 실패 메커니즘):}
분류군 A 의 실패는 critical-path H2D/D2H 동기화 비용, 분류군 C 의 실패는 sub-step scheduling 의 lock contention 으로 설명된다.
검증: 분류군 A 의 대표 4 기법 (AVX-512 tokenizer / sampling, AMX matmul / prefill), 분류군 C 의 대표 2 기법 (task-pool stub / dummy fill), 분류군 D 의 대표 2 기법 (cold-KV decompress, AMX draft proxy).

\subsection{Paired Control/Treatment 측정 protocol}
\label{subsec:protocol}

각 후보 기법 $L$ 에 대해 다음 paired 측정을 수행한다.

\textbf{대조군 (Control, OFF)}: \S\ref{subsec:agsd} 의 적응형 워크로드 라우팅 기준 구성, 기법 $L$ 비활성.
\textbf{처리군 (Treatment, ON)}: 기준 구성 + 기법 $L$ 활성, 다른 모든 인자 동일.

각 모드에서 3-mix 워크로드 (balanced/sonnet-heavy/code-heavy) $\times$ 3 backend scenario (baseline-only / spec-decode-only / adaptive-routed) 를 측정한다.
주요 메트릭은 throughput (tokens/s), wall-clock time (s), latency p50/p99 (ms) 이다.
기법 $L$ 의 단독 효과 $\Delta(L)$ 은 ON 과 OFF 의 처리량 비율 변화로 정의한다.

본 연구의 \emph{유의 임계점}은 $\Delta(L) \geq 5\,\%$ 로 설정한다.
이는 1-run 측정 분산이 $\pm 1$--$3\,\%$ 인 환경에서 신호 vs.\ noise 의 보수적 경계값이다.

\subsection{실험 환경}
\label{subsec:setup}

본 연구의 모든 측정은 다음 통일된 환경에서 수행되었다.
모델: Qwen 2.5-32B-Instruct (bf16).
GPU: NVIDIA H100 8장.
CPU: Intel Xeon SPR-class, 듀얼 NUMA 노드 (각 노드 56 phys cores, distance 비 2.1$\times$).
인스턴스: 텐서 병렬도 4 $\times$ 2 인스턴스 (baseline backend $\rightarrow$ GPU 0--3, spec-decode backend $\rightarrow$ GPU 4--7).
\verb|max_model_len| 4{,}096, \verb|max_num_seqs| 128, \verb|max_num_batched_tokens| 4{,}096.
워크로드: 500-prompt 3-mix, max\_tokens 32, concurrency 32.
Spec config (treatment backend): SuffixDecoding~\cite{snowflake-arctic}, $K=32$.
cudagraph: PIECEWISE 모드.

\subsection{Multi-Run 분산 protocol}
\label{subsec:multirun}

핵심 후보 기법 (NUMA pin, regular auxiliary 100\,ms, async tokenizer 20\,ms) 에 대해 3회 반복 측정을 수행하여 분산을 확인한다.
첫 번째 실행은 cudagraph 컴파일 진행 중인 initialization-phase outlier 가능성이 있어, \emph{steady-state} (2회/3회의 평균) 과 \emph{multi-run mean} (3회 평균) 을 별도로 보고한다 (Table~\ref{tbl:multirun}).
1-run 측정에서 부호가 반전될 수 있는 magnitude $< 3\,\%$ 신호는 단독으로 채택하지 않는다.
